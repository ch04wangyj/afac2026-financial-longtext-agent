# 2026-06-09 收敛版结论

| 类别 | 当前决策 | 原因 |
|------|----------|------|
| 默认主线 | V1 → V2 → V3 条件触发 | 先保证合规可复现，再提升准确率 |
| 检索核心 | bm25s / BM25+ / 字符 2-4 gram / 字段精确索引 | 无 embedding，速度和可解释性更好 |
| 压缩核心 | 规则抽取式压缩 + KVP/表格 JSON | 零 Token、保留原文证据、规避非 Qwen 模型争议 |
| 小模型压缩 | LLMLingua 等仅作实验，不进默认链路 | 代码审核时可能被解释为非 Qwen 模型能力 |
| FactCard | 只允许规则生成免费；Qwen 生成必须计入 Token | 题面只明确文档解析不计 Token，LLM 摘要不能默认免费 |
| 自检策略 | 低置信触发 CoVe / Self-Consistency | 全题触发 Token 成本过高 |
| 模型选择 | qwen3.7-plus 主模型；qwen3.7-max 做难题/冲榜复核；低成本关键词只在规则允许时测试 | 与当前代码和百炼 OpenAI-compatible API 配置一致 |
| RAG 框架扩展 | 保留 BM25/RRF 主路；纳入 LogicRAG-lite、CRAG-lite；Graph/Linear 先做实验分支 | 详见 `AFAC2026_赛题四_RAG框架扩展路线图.md`，避免直接套用违反 embedding/模型约束的原框架 |

# Stage 0: PDF 解析与预处理

> **约束**: 离线阶段，Token 消耗不计入预算。可以使用任何 Python 库/工具。

## 0.1 PDF 文本提取

### 0.1.1 PyMuPDF (fitz)
- **是什么**: 基于 MuPDF 的高性能 PDF 库，提取文本、图片、元数据
- **为什么用**: C 内核极快（比 pdfplumber 快 3-10×）；对电子版 PDF 文本提取准确率最高（DocLayNet 基准第一）；支持页面级随机访问
- **为什么不用**: 对扫描件/图片型 PDF 无法提取文本（需要 OCR 管道）；表格提取能力弱于 pdfplumber；中文竖排文本支持有限
- **赛题适用性**: ✅ **推荐** — insurance/regulatory/research 领域首选，因其 PDF 多为电子版

### 0.1.2 pdfplumber
- **是什么**: 基于 pdfminer.six 的 PDF 解析库，精确控制字符级坐标
- **为什么用**: 表格提取能力强于 PyMuPDF；字符级坐标精确（可还原复杂版面）；支持可视化调试
- **为什么不用**: 速度慢（特别是 300+ 页的 contracts PDF）；内存占用高；对简单电子版 PDF 过度设计
- **赛题适用性**: ✅ **推荐** — financial_contracts 领域首选，因其含大量债券条款表格

### 0.1.3 PyPDF2 / pypdf
- **是什么**: 纯 Python 的 PDF 读写库，轻量级
- **为什么用**: 极轻量，依赖最少；支持 PDF 合并/分割/元数据操作
- **为什么不用**: 文本提取质量不如 PyMuPDF；不支持表格提取；对复杂版面处理能力弱
- **赛题适用性**: 🟡 **可选** — 仅用于 PDF 元数据提取和文件合并拆分

### 0.1.4 MinerU (PDF → Markdown)
- **是什么**: 上海 AI Lab 开源的 PDF 转 Markdown 工具，集成版面分析 + OCR + 公式识别
- **为什么用**: 对扫描件 PDF 效果优秀（自动检测文本/图片区域）；输出 Markdown 保留结构（标题/表格/公式）；专为 LLM 下游消费设计
- **为什么不用**: 需 GPU（版面分析模型）；处理速度慢（300 页财报可能需要几分钟）；依赖较多（Magic-PDF/Magic-Doc）
- **赛题适用性**: ✅ **推荐** — financial_reports 领域首选（扫描件 OCR）

### 0.1.5 Docling (IBM)
- **是什么**: IBM 开源的文档理解工具，PDF → 结构化 JSON/Markdown
- **为什么用**: 最新的版面分析模型（DocLayNet-v2）；对复杂多栏版面效果好；输出格式丰富
- **为什么不用**: 较新项目（2024 年发布）；处理速度待优化；中文长文档测试较少
- **赛题适用性**: 🟡 **可选** — 备选方案，如果 MinerU 效果不佳可尝试

### 0.1.6 Marker (Surya-based)
- **是什么**: 基于 Surya OCR 的 PDF → Markdown 转换工具
- **为什么用**: 对复杂公式和表格效果好；支持多语言（含中文）；GPU 加速
- **为什么不用**: 需要 GPU；中文 PDF 的测试社区较小；对 300+ 页文档可能 OOM
- **赛题适用性**: 🟡 **可选** — 主要用于需要公式识别的 research PDF

### 0.1.7 Unstructured.io
- **是什么**: 企业级文档预处理框架，支持多种文件格式
- **为什么用**: 统一 API 处理 PDF/HTML/Word 等格式；内置分块和版面分析；Partition 功能强大
- **为什么不用**: API 模式需付费（开源版功能受限）；对中文金融术语的分块可能需额外调参；笨重
- **赛题适用性**: 🟡 **可选** — 如果需要统一处理多种格式（非必须，赛题基本是 PDF）

### 0.1.8 GROBID
- **是什么**: 学术文献的结构化解析工具（标题/作者/引用/章节）
- **为什么用**: 学术级 PDF 结构解析最佳；输出 TEI XML 保留完整结构
- **为什么不用**: 针对学术论文优化，不适合金融文档（保险合同/法规/财报）；需要 Java 运行时
- **赛题适用性**: ⚠️ **慎用** — 仅对 research 领域的研报 PDF 有价值

### 0.1.9 pdf2htmlEX
- **是什么**: PDF → HTML 转换，保留精确的文本位置和字体
- **为什么用**: HTML 输出保留精确坐标，便于后续 CSS 选择器提取；视觉效果完美还原
- **为什么不用**: 不对金融文档做语义理解；输出 HTML 体积大
- **赛题适用性**: ❌ **不可用** — 不如直接用 MinerU/Docling 的结构化输出

## 0.2 OCR 引擎

### 0.2.1 PaddleOCR
- **是什么**: 百度开源的中文 OCR 引擎，PP-OCRv4 版本
- **为什么用**: 中文识别率最优（开源方案中）；支持表格结构识别（PP-Structure）；可离线部署；轻量模型可选（75% 体积压缩）
- **为什么不用**: 对超长文档的处理需要分批；GPU 加速可选但非必须（CPU 也可运行）
- **赛题适用性**: ✅ **推荐** — 财报 OCR 的首选方案

### 0.2.2 Tesseract (with Chinese tessdata)
- **是什么**: Google 维护的经典 OCR 引擎，5.x 支持 LSTM
- **为什么用**: 老牌成熟稳定；社区庞大
- **为什么不用**: 中文识别率远低于 PaddleOCR（需要大量调参）；表格识别需外挂；对扫描质量敏感
- **赛题适用性**: ⚠️ **慎用** — 仅在 PaddleOCR 不可用时做备选

### 0.2.3 Surya OCR
- **是什么**: 基于 Transformer 的新一代 OCR，支持 90+ 语言
- **为什么用**: 版面分析和文本识别一体化；对复杂排版效果好；GPU 加速
- **为什么不用**: 需 GPU；中文长文档的测试数据较少
- **赛题适用性**: 🟡 **可选** — 需要 GPU，结合 Marker 使用

### 0.2.4 EasyOCR
- **是什么**: 基于 PyTorch 的通用 OCR，支持 80+ 语言
- **为什么用**: 简单易用，pip install 即可
- **为什么不用**: 中文识别率不如 PaddleOCR；速度较慢（CPU 上尤其）
- **赛题适用性**: ⚠️ **慎用** — PaddleOCR 在中文上更优

### 0.2.5 DeepSeek-OCR (VLM-based)
- **是什么**: 基于视觉语言模型(VLM)的新一代 OCR，直接输出结构化 JSON/Markdown
- **为什么用**: 2025 最新；Token 压缩技术降低下游成本；自带版面理解
- **为什么不用**: 需 GPU（VLM 推理）；项目较新、社区较小；**可能被视为"embedding 模型"的边界**（需判断是否违规）
- **赛题适用性**: ⚠️ **慎用** — VLM 可能触及赛题约束边界，需确认是否算"其他模型"

### 0.2.6 阿里云 OCR API
- **是什么**: 阿里云提供的商业 OCR 服务
- **为什么用**: 精度高；按量付费；支持发票/财报等场景
- **为什么不用**: 需联网；有费用；**但离线阶段不计赛题 Token！**
- **赛题适用性**: 🟡 **可选** — 如果自建 OCR 精度不够

## 0.3 表格提取

### 0.3.1 Camelot
- **是什么**: 专攻 PDF 表格提取的 Python 库，基于 Lattice（线框检测）和 Stream（空白检测）两种模式
- **为什么用**: 对有线框表格（金融合同中的债券条款表）效果极好；输出 pandas DataFrame
- **为什么不用**: 对扫描件无效（需先 OCR）；对无线框表格（Stream 模式）精度不高；速度慢
- **赛题适用性**: ✅ **推荐** — financial_reports 表格提取（结合 OCR 使用）

### 0.3.2 Tabula
- **是什么**: 基于 Java 的 PDF 表格提取工具，有 Python 封装
- **为什么用**: 对简单表格效果好；交互式 GUI 可用于调试
- **为什么不用**: 需 Java 运行时；精度不如 Camelot；已较少维护
- **赛题适用性**: ⚠️ **慎用** — 逐步被 Camelot 替代

### 0.3.3 pdfplumber 内置表格
- **是什么**: pdfplumber 自带的 `extract_table()` / `extract_tables()`
- **为什么用**: 与文本提取共用管道；对无边界框的表格有一定识别能力
- **为什么不用**: 对复杂表格（合并单元格/嵌套表）支持有限
- **赛题适用性**: ✅ **推荐** — contracts 中将 pdfplumber 文本+表格一并提取

### 0.3.4 PaddleOCR PP-Structure
- **是什么**: PaddleOCR 内置的表格结构识别模块
- **为什么用**: 端到端：OCR → 版面分析 → 表格结构 → HTML/Excel 输出；中文表格支持好
- **为什么不用**: 需 GPU 加速（CPU 很慢）；对超长宽表可能截断
- **赛题适用性**: ✅ **推荐** — OCR 流程中的表格提取

## 0.4 版面分析

### 0.4.1 LayoutParser
- **是什么**: 基于 Detectron2 的版面分析库，可识别标题/正文/图片/表格/公式
- **为什么用**: 模块化设计，可训练自定义模型；预训练模型覆盖常见版面元素
- **为什么不用**: 需 GPU；模型较大；Python 环境配置复杂
- **赛题适用性**: 🟡 **可选** — 如果需要精细的版面分割

### 0.4.2 DocLayout-YOLO
- **是什么**: 基于 YOLO 的文档版面检测，速度快
- **为什么用**: YOLO 架构推理极快（比 LayoutParser 快 10×+）；精度接近 Detectron2 方案
- **为什么不用**: 需要 GPU 训练/推理；中文文档预训练模型较少
- **赛题适用性**: 🟡 **可选** — 快速版面分割方案

### 0.4.3 MinerU 内置版面分析
- **是什么**: MinerU 自带的版面分析模型（基于 LayoutReader）
- **为什么用**: 与 PDF→Markdown 管道深度融合；无需单独配置
- **为什么不用**: 黑白盒，难以单独调优
- **赛题适用性**: ✅ **推荐** — 使用 MinerU 时无需额外方案

# Stage 1: 文本分块与分段

> **核心问题**: 怎么切？切多大？切完之后怎么组织？
> **约束**: 离线阶段，不计 Token。但分块质量直接影响后续检索效果。

## 1.1 分块策略

### 1.1.1 固定大小分块 (Fixed-size)
- **是什么**: 按固定 token 数切割，如每 512 token 一块，重叠 50 token
- **为什么用**: 最简单、最可控；所有文档统一处理；实现成本为零
- **为什么不用**: 无视文档语义边界——可能在句子/段落/表格中间切断；金融条款的逻辑单元被打破
- **赛题适用性**: ⚠️ **慎用** — 仅用于基线快速验证，不应作为最终方案

### 1.1.2 递归字符分割 (Recursive Character)
- **是什么**: 按优先级嵌套分割符逐步切割：`\n\n` → `\n` → `。` → ` ` → ``
- **为什么用**: 比固定大小更语义友好；LangChain 默认方案，开箱即用
- **为什么不用**: 仍然不理解文档逻辑结构；对中文的句号/分号分割效果一般
- **赛题适用性**: 🟡 **可选** — 作为快速实现基线

### 1.1.3 句子级分割 (Sentence-level)
- **是什么**: 按标点符号（句号、分号、问号）将文本分割为独立句子
- **为什么用**: 句子是自然语言的最小语义单元；对保险条款中的单句规定效果好
- **为什么不用**: 金融文档中的长句可能跨多个条件——一句话包含产品A/B/C的对比
- **赛题适用性**: 🟡 **可选** — 作为更粗粒度分块的补充

### 1.1.4 段落级分割 (Paragraph-level)
- **是什么**: 按空行/换行符分割为段落
- **为什么用**: 段落是自然的信息单元；对研报/法规效果好
- **为什么不用**: 保险合同中的大段条款可能被切成一块，粒度太粗；表格被破坏
- **赛题适用性**: 🟡 **可选** — 适合 research 和 regulatory 领域

### 1.1.5 文档结构感知分块 (Structure-aware) ⭐
- **是什么**: 基于文档的逻辑结构切分：标题层级（H1/H2/H3）、条款编号（第X条/6.1）、表格边界
- **为什么用**: 保留金融文档最关键的结构信息；条款作为最小检索单元逻辑自洽；避免切断表格
- **为什么不用**: 需要为每种文档类型编写特定的解析规则；实现成本较高
- **赛题适用性**: ✅ **推荐** — 这是赛题的核心竞争力之一，每个领域的实现不同：
  - insurance: 按「第X条」/「X.X」编号切分
  - regulatory: 按「第X条」/「第X章」切分
  - contracts: 按章节标题 + 表格边界切分
  - reports: 按表格 + 章节混合切分
  - research: 按标题层级切分

### 1.1.6 条款级分块 (Clause-level)
- **是什么**: 针对保险/法规文档，以条款编号（"第X条"）为分割边界
- **为什么用**: 每个条款是一个独立的权利/义务/条件单元；检索时精确匹配条款编号；法律逻辑自洽
- **为什么不用**: 只适用于保险和法规领域（contracts/research/reports 不完全适用）
- **赛题适用性**: ✅ **推荐** — insurance + regulatory 核心分块策略

### 1.1.7 表格优先分块 (Table-first)
- **是什么**: 先检测和提取所有表格（每个表格独立成块），剩余文本再按段落/章节分块
- **为什么用**: 财报中 80% 的信息在表格中（营收/利润/现金流）——表格的语义单元是行+列，不是文字流
- **为什么不用**: 只适用于财务报表类文档
- **赛题适用性**: ✅ **推荐** — financial_reports 核心分块策略

### 1.1.8 语义分块 (Semantic Chunking)
- **是什么**: 计算相邻句子的 embedding 相似度，在语义断裂点（相似度低）处切分
- **为什么用**: 理论上最符合语义边界；对叙述性文本（研报）效果好
- **为什么不用**: **需要 embedding 模型 → 违反赛题规则！** 🚫
- **赛题适用性**: ❌ **不可用** — 违反 Embedding 禁令

### 1.1.9 代理式分块 (Agentic Chunking)
- **是什么**: 让 LLM 决定分块边界——给 LLM 一段文本，让它标注最佳切割位置
- **为什么用**: 最语义化的分块策略；LLM 可以理解上下文判断自然断点
- **为什么不用**: 消耗大量 Qwen Token（每段文本都需一次 LLM 调用）；处理 190+ PDF 的成本极高
- **赛题适用性**: ⚠️ **慎用** — 仅供特定核心文档（如发现某篇 PDF 结构极复杂），不建议全量使用

### 1.1.10 多粒度分块 (Multi-granularity / Hierarchical)
- **是什么**: 同时维护多种粒度的分块：细粒度（条款级 200-500 token）+ 粗粒度（章节级 1000-2000 token）+ 文档级规则画像（DocProfile/KVP）
- **为什么用**: 简单问题用细粒度检索 → 精确命中；复杂问题用粗粒度 → 获取上下文；跨文档比较用文档画像 → 覆盖面广
- **为什么不用**: 维护多套索引增加存储和实现复杂度；检索结果需要跨粒度融合
- **赛题适用性**: ✅ **推荐** — V2/V3 的核心设计

### 1.1.11 Small-to-Big / 父子分块
- **是什么**: 检索时用小块（高精度），返回时送回大块（完整上下文）。即：用子块做检索匹配，返回父块的完整内容
- **为什么用**: 解决「检索精度」和「上下文完整性」之间的矛盾——小块检索更精确，大块提供完整推理上下文
- **为什么不用**: 需要维护父子映射关系；实现稍复杂
- **赛题适用性**: ✅ **推荐** — 简单有效的增强策略

### 1.1.12 滑动窗口分块
- **是什么**: 固定窗口大小 + 固定步长滑动，相邻块有大量重叠
- **为什么用**: 确保边界信息不丢失；简单实现
- **为什么不用**: 大量冗余（重叠部分被重复索引）；检索效率低
- **赛题适用性**: ⚠️ **慎用** — 仅作为其他分块策略的补充（在关键位置的 sentence window 扩展）

## 1.2 分块元数据

每个块应附加以下元数据字段（离线阶段提取，不耗 Token）：

| 元数据字段 | 用途 | 提取方式 |
|-----------|------|----------|
| `chunk_id` | 唯一标识 | 自动生成 |
| `doc_id` | 来源文档 | 文件名映射 |
| `doc_title` | 文档标题 | PDF 元数据/首页提取 |
| `domain` | 所属领域 | 目录名 |
| `chunk_type` | 块类型（clause/text/table/figure） | 解析时标注 |
| `section_title` | 所在章节标题 | 结构解析提取 |
| `clause_id` | 条款编号（如"第47条"） | 正则提取 |
| `page_range` | 页码范围 | PDF 页码 |
| `entity_mentions` | 实体提及（产品名/公司名/法条名） | NER 提取 |
| `numeric_range` | 涉及数值范围 | 数值提取 |
| `prev_chunk_id` / `next_chunk_id` | 前后块链接 | 自动生成 |

# Stage 2: 中文文本处理

> **核心问题**: BM25 的词法匹配极度依赖分词质量。中文没有天然的词边界，不同的分词策略导致完全不同的检索效果。

## 2.1 分词方案

### 2.1.1 jieba 默认分词
- **是什么**: 基于前缀词典和 HMM 的中文分词器
- **为什么用**: 最成熟、最快的 Python 中文分词器；开箱即用；社区庞大
- **为什么不用**: 默认词典不含金融术语（"身故保险金"→"身故/保险金"？"退保费用率"→"退保/费用/率"？）；金融词汇切分不准
- **赛题适用性**: 🟡 **可选** — 仅基线使用

### 2.1.2 jieba + 自定义金融词典 ⭐
- **是什么**: 在 jieba 基础上加载金融领域词典，强制金融术语不被切分
- **为什么用**: 保留 jieba 速度，大幅提升金融术语准确率；词典可以从 PDF 中自动抽取（高频多字词组）
- **为什么不用**: 需要手动维护词典（但可以从 PDF 中自动提取）；新词/组合词仍可能被切分
- **赛题适用性**: ✅ **推荐** — 实现简单、效果显著

### 2.1.3 字符 n-gram (Char n-gram)
- **是什么**: 将文本切割为 n 个字符的片段，如 "平安智盈金生" → ["平安", "安智", "智盈", "盈金", "金生"] (2-gram) + ["平安智", "安智盈", ...] (3-gram)
- **为什么用**: **完全不需要词典！** 天然覆盖所有子串组合——"身故保险金"即使被 jieba 切错，char n-gram (2-4) 也能用子串匹配命中；对错别字和变体表述鲁棒（金融文档的同义词替换）
- **为什么不用**: 索引体积增大（n-gram 数量远多于词数）；可能引入噪音匹配（过短的子串）
- **赛题适用性**: ✅ **推荐** — **最安全的方案**，不依赖词典质量，在金融术语匹配上反而更优

### 2.1.4 jieba + 字符 n-gram 混合
- **是什么**: 同时使用词级（jieba+词典）和字符级（n-gram）两种分词，分别建索引，RRF 融合
- **为什么用**: 取两者之长——词典分词精确，n-gram 覆盖全面（防止遗漏）；互补性强
- **为什么不用**: 需要维护两套索引；RRF 融合引入额外复杂度
- **赛题适用性**: ✅ **推荐** — 方案二/三的核心设计

### 2.1.5 pkuseg
- **是什么**: 北大开源分词器，支持领域适应
- **为什么用**: 分词精度高于 jieba（在多个基准上）；支持微调领域模型（如果有标注数据）
- **为什么不用**: 速度慢于 jieba（约 3-5×）；金融领域预训练模型不可用（赛题禁止微调/加载其他模型）
- **赛题适用性**: ⚠️ **慎用** — 无金融预训练模型时，优势有限

### 2.1.6 THULAC / LTP
- **是什么**: 清华/哈工大的中文 NLP 工具包，含分词+词性标注+NER
- **为什么用**: 附带词性标注和 NER，可辅助实体提取
- **为什么不用**: 速度慢（比 jieba 慢 10×+）；安装复杂；金融领域未优化
- **赛题适用性**: ⚠️ **慎用** — 仅在需要词性标注辅助实体提取时考虑

### 2.1.7 基于 Qwen 的动态分词
- **是什么**: 让 Qwen 对查询做分词/关键词提取，将"中文分词"问题转化为"LLM 关键词提取"
- **为什么用**: Qwen 对中文的语义理解远超任何规则分词器；可直接跨越分词歧义
- **为什么不用**: 每次查询消耗 Token（~100-200 token）
- **赛题适用性**: 🟡 **可选** — 不是替代 BM25 的分词，而是作为查询端的关键词生成

## 2.2 文本预处理

### 2.2.1 停用词过滤
- **是什么**: 移除「的」「了」「在」「是」等高频无义虚词
- **为什么用**: 减少索引噪音，提高检索精度
- **为什么不用**: 在字符 n-gram 模式下，停用词过滤可能导致子串断裂；BM25 本身的 IDF 已经能压制高频词
- **赛题适用性**: 🟡 **可选** — 在词级索引上使用；n-gram 模式下不建议

### 2.2.2 全半角转换 + 数字归一化
- **是什么**: 全角数字→半角（１２３→123）；中文数字→阿拉伯数字（一百二十万→120万）；统一日期格式
- **为什么用**: 金融文档中数字是关键检索对象，不同文档的写法差异导致匹配失败
- **为什么不用**: 需要小心处理——有些"十二"是编号不是数字
- **赛题适用性**: ✅ **推荐** — 数字归一化对计算题至关重要

### 2.2.3 繁体→简体转换
- **是什么**: 繁体中文→简体中文的统一转换
- **为什么用**: 部分法规/研报可能含繁体内容
- **为什么不用**: 如果所有 PDF 已经是简体，则不需要
- **赛题适用性**: 🟡 **可选** — 抽样检查数据后决定

### 2.2.4 正则保护 (Protected Patterns)
- **是什么**: 压缩/分块前用正则标记关键信息行（金额、日期、百分比、条款编号），确保不被破坏
- **为什么用**: 金融 QA 的错误往往源于某个数值/条款编号在压缩过程中丢失
- **为什么不用**: 实现简单，无明显代价
- **赛题适用性**: ✅ **推荐** — 零成本高质量保护

# Stage 3: 索引构建与检索

> **核心约束**: 🚫 禁止使用 Embedding 模型。所有检索必须基于词法/统计/规则/LLM 方法。
> **关于 SPLADE/ColBERT 的边界判定**: SPLADE 使用 BERT 预测词权重，ColBERT 生成 token 级 embedding。由于赛题禁止使用"embedding 模型"且仅允许 Qwen 系列 API，这些依赖其他神经网络模型的方法均被视为**违规**。

## 3.1 词法检索模型（核心引擎）

### 3.1.1 BM25 (Okapi) — 基准
- **是什么**: 概率检索框架下的经典排序函数，TF 饱和 + 文档长度归一化 + IDF
- **为什么用**: 无需任何模型；对金融术语天生敏感（精确词匹配）；几十年工业验证；rank_bm25 一行代码
- **为什么不用**: 无语义理解（"身故" vs "死亡" ≈ 0 匹配）；无法处理同义词/近义词；对查询的措辞敏感
- **赛题适用性**: ✅ **推荐** — 所有方案的检索核心

### 3.1.2 BM25+ (Lower-bound Correction)
- **是什么**: BM25 的修正版，为所有文档添加一个小的正数补偿项 δ，解决 BM25 对超长文档的过度惩罚
- **为什么用**: 原始 BM25 对极长文档（如 300 页的 contracts）惩罚过重——即使包含相关条款，得分也很低；BM25+ 缓解这个问题
- **为什么不用**: 额外参数 δ 需要调优；在中等长度文档集上改善有限
- **赛题适用性**: ✅ **推荐** — contracts 和 reports 领域使用 BM25+

### 3.1.3 BM25L
- **是什么**: 改进 BM25 的文档长度归一化，减少对长文档的不公平惩罚
- **为什么用**: 与 BM25+ 类似但通过重写长度归一化公式实现；对极长文档效果更好
- **为什么不用**: 参数调优更复杂；与 BM25+ 选其一即可
- **赛题适用性**: 🟡 **可选** — 如果 BM25+ 效果不理想可尝试

### 3.1.4 BM25Adpt
- **是什么**: 自适应的 BM25——根据查询特征动态调整 k1 参数（短查询 vs 长查询）
- **为什么用**: 不同类型问题（"什么是X" vs "比较ABCD四个产品的身故保险金计算"）的最优 k1 不同；动态调整优于固定参数
- **为什么不用**: 需要查询分类器（额外组件）；实现较复杂
- **赛题适用性**: 🟡 **可选** — V3 可使用

### 3.1.5 BM25F (Field-based)
- **是什么**: BM25 扩展到多字段——每个字段（标题/条款号/正文/表格）有独立权重，最后线性组合
- **为什么用**: 金融文档的字段结构非常明确：条款编号的匹配比正文匹配更重要；标题匹配比正文匹配更相关
- **为什么不用**: 需要预定义字段结构（但我们的分块元数据已经有这些字段）
- **赛题适用性**: ✅ **推荐** — 将 clause_id/entity/title 作为独立字段加权

### 3.1.6 BM25T / BM25T+ (for Table)
- **是什么**: 针对表格数据的 BM25 变体——将「表头+对应行」作为文档单元，保留表头信息
- **为什么用**: 财报表格中"2025营业收入：1,200亿"——如果表头被切掉，"1,200亿"就失去了意义
- **为什么不用**: 仅对表格型文档有效
- **赛题适用性**: 🟡 **可选** — financial_reports 专属优化

### 3.1.7 BMX (BM𝔛)
- **是什么**: Li et al. (2024) 提出的 BM25 集成框架，将多种 BM25 变体的分数 ensemble
- **为什么用**: 组合多种变体的优势；鲁棒性更强
- **为什么不用**: 计算开销增加（多种评分函数并行）；参数空间增大
- **赛题适用性**: 🟡 **可选** — 冲榜阶段尝试

### 3.1.8 查询侧 BM25 (BM25Q)
- **是什么**: 双向 BM25——不仅考虑文档侧词频，还考虑查询侧词频和长度
- **为什么用**: 本赛题的查询（Qwen 生成的关键词组）长度不一（2-20词），不同长度查询需要不同的 TF 处理
- **为什么不用**: 增加实现复杂度；与 BM25+ 组合可能过度
- **赛题适用性**: 🟡 **可选**

## 3.2 经典概率模型（BM25 的替代品）

### 3.2.1 TF-IDF (Vector Space Model)
- **是什么**: 经典词频-逆文档频率，余弦相似度
- **为什么用**: 是最简单的 baselines；可完全用 scikit-learn 一行实现
- **为什么不用**: 各方面均弱于 BM25（无 TF 饱和、无文档长度归一化）
- **赛题适用性**: ⚠️ **慎用** — 仅用于 baseline 对比

### 3.2.2 DFR (Divergence from Randomness) — PL2
- **是什么**: 基于随机性偏离的检索模型族——如果一个词在一篇文档中出现的频率显著偏离随机分布，则它是重要的
- **为什么用**: 在某些 TREC 任务中超越 BM25；对不同类型的词分布自动适应
- **为什么不用**: 在中文金融文档上的测试数据缺乏；Terrier 工具依赖（Java）
- **赛题适用性**: 🟡 **可选** — 理论研究价值，工程代价较高

### 3.2.3 Language Model with Dirichlet Smoothing
- **是什么**: 统计语言模型方法——P(Q|D) ∝ ∏ P(t|D)，用 Dirichlet 先验平滑
- **为什么用**: 平滑机制对低频金融术语友好（不至于零概率）
- **为什么不用**: 与 BM25 在大多数场景性能接近；实现复杂度高
- **赛题适用性**: 🟡 **可选** — 如果你的 IR 背景偏好 LM 方法

### 3.2.4 Language Model with Jelinek-Mercer Smoothing
- **是什么**: JM 平滑的 LM 方法——用 λ 参数混合文档模型和集合模型
- **为什么用**: λ 参数可调，对不同领域灵活
- **为什么不用**: Dirichlet 平滑通常更好
- **赛题适用性**: ⚠️ **慎用** — 一般不如 Dirichlet

### 3.2.5 In_expB2 / In_expC2 / InL2 (DFR 变体)
- **是什么**: DFR 框架下的不同标准化模型（B2/C2/L2）
- **为什么用**: 不同模型对不同类型的文本（条款 vs 叙述）可能有差异化效果
- **为什么不用**: 绝大多数场景 BM25 足够好；中文金融领域无对比数据
- **赛题适用性**: ⚠️ **慎用** — 仅当 BM25 在特定领域效果不佳时探索

## 3.3 多字段/多路检索架构

### 3.3.1 单索引 + 统一检索
- **是什么**: 所有文本块放在一个 BM25 索引中，无领域/字段区分
- **为什么用**: 最简单；题目给出 doc_ids（A 榜），无需文档检索
- **为什么不用**: 跨领域噪音（保险查询匹配到法规条款）；无法利用结构化信息
- **赛题适用性**: ⚠️ **慎用** — 仅基线使用

### 3.3.2 领域隔离索引 ⭐
- **是什么**: 每个领域（insurance/regulatory/contracts/reports/research）维护独立 BM25 索引
- **为什么用**: A 榜题目给出 domain 字段，可直接路由到正确索引；避免跨领域噪音；每个索引可用领域最优 BM25 参数
- **为什么不用**: B 榜盲搜时可能跨领域出题——需要先判断领域
- **赛题适用性**: ✅ **推荐** — A 榜用 domain 路由，B 榜用 Qwen 判断领域

### 3.3.3 多字段独立索引 ⭐
- **是什么**: 同时维护多个独立索引：全文 BM25 + 条款号精确索引(Hash) + 实体精确索引(Aho-Corasick) + 数值范围索引 + 表格索引
- **为什么用**: 不同查询需求用不同索引——条款号查询走精确索引，数值范围走过滤索引，全文模糊走 BM25；各索引互补
- **为什么不用**: 维护成本和存储增加（但离线阶段不计 Token）
- **赛题适用性**: ✅ **推荐** — 方案二~六的核心设计

### 3.3.4 文档级索引 (Doc-level)
- **是什么**: 不以 chunk 为检索单元，而以文档（整个 PDF）为单元评分排序
- **为什么用**: B 榜需要先确定相关文档再精细检索
- **为什么不用**: 粗粒度排序丢失细粒度信息
- **赛题适用性**: ✅ **推荐** — B 榜第一级粗筛（文档 → Chunks）

### 3.3.5 跨领域统一索引 (for B 榜)
- **是什么**: A 榜按 domain 路由；B 榜没有 domain 信息，需要一个跨所有领域的统一索引做第一轮检索
- **为什么用**: B 榜必须自行判断相关文档
- **为什么不用**: A 榜不需要
- **赛题适用性**: ✅ **推荐** — B 榜专属索引

## 3.4 查询处理策略

### 3.4.1 Qwen 关键词提取（核心）⭐
- **是什么**: 题目 → Qwen 提取关键实体 + 生成 3-5 组搜索关键词 → 多组并行检索
- **为什么用**: Qwen 的语义理解远超任何规则方法；可以识别隐式查询意图（"比较A和B" → 单独搜A、单独搜B、搜AB共同点）；每查询仅 ~200 token
- **为什么不用**: 需要 Token（但很少）
- **赛题适用性**: ✅ **推荐** — V1/V2/V3 的查询入口

### 3.4.2 IterKey 迭代关键词生成 ⭐
- **是什么**: 第一轮检索结果不足 → Qwen 分析缺什么 → 生成新关键词 → 第二轮检索（最多 2-3 轮）
- **为什么用**: 显著提升召回（+5~20% 准确率）；探索式检索：第一轮不知道缺什么，看了结果才知道
- **为什么不用**: 每次迭代需要额外的 Qwen Token（~300 token/轮）；迭代次数需控制（最多 2-3 轮）
- **赛题适用性**: ✅ **推荐** — V2/V3 的核心机制

### 3.4.3 ELITE 式查询缓存
- **是什么**: 同文档题目共享关键词缓存——题目 1 搜了"平安智盈金生 身故保险金"，题目 2 涉及同一产品时直接复用
- **为什么用**: 显著减少查询阶段的 Token（同文档多题时可省 30-50%）
- **为什么不用**: 需要题目分组逻辑（先按文档分组再批量处理）
- **赛题适用性**: ✅ **推荐** — V3 的批处理核心

### 3.4.4 伪相关反馈 (Pseudo Relevance Feedback, PRF) ⭐
- **是什么**: BM25 检索 Top-K → 从 Top-K 提取高 IDF 词作为扩展词 → 用原查询+扩展词重新检索
- **为什么用**: **完全免费（零 Token）** 的查询扩展；SIGIR 2024 实验证实 PRF 超越纯 BM25；对长文档集效果尤其好
- **为什么不用**: 如果 Top-K 的文档是噪音（假相关），扩展词会引入噪音 → 检索质量恶化
- **赛题适用性**: ✅ **推荐** — 所有方案的默认增强

### 3.4.5 相关性模型 (Relevance Model RM3)
- **是什么**: PRF 的高级版本——用统计语言模型估计相关文档中的词分布，而非简单的 TF-IDF 选词
- **为什么用**: 比简单 PRF 更精确的词选择；理论基础更强
- **为什么不用**: 计算复杂度高于 PRF；需要 Terrier 或自行实现
- **赛题适用性**: 🟡 **可选** — 如果 PRF 效果不够好

### 3.4.6 Qwen 查询改写
- **是什么**: 题目的自然语言描述 → Qwen 重写为更适合 BM25 检索的关键词组合
- **为什么用**: 原始题目措辞可能不适合词法检索（口语化、指代不明确）；Qwen 可以分解复杂问题为原子查询
- **为什么不用**: 每次查询需 ~200 Token
- **赛题适用性**: ✅ **推荐** — 查询阶段的必要 Token 支出

### 3.4.7 多查询融合 (Multi-Query Fusion)
- **是什么**: 对同一问题生成多个查询变体 → 分别检索 → 结果融合（提升覆盖）
- **为什么用**: 不同措辞命中不同相关文档；提升召回
- **为什么不用**: 检索次数增加（总检索量 = 变体数 × 单次检索）
- **赛题适用性**: 🟡 **可选** — V2+ 使用

### 3.4.8 子问题分解 (Query Decomposition)
- **是什么**: 复杂问题（"比较A/B/C/D四个产品"）分解为原子子问题（"A产品的X条款是什么"）
- **为什么用**: 复杂问题的不同部分需要在不同文档中定位；分解后每个子问题指向明确
- **为什么不用**: 需要多次 Qwen 调用；子问题之间的答案需要合并
- **赛题适用性**: ✅ **推荐** — 多文档比较题的必需策略

### 3.4.9 Step-Back Querying
- **是什么**: 先问更抽象/更一般的问题，再问具体问题（"什么是身故保险金的计算规则？" → "各产品具体是多少？"）
- **为什么用**: 对不知道从哪开始的复杂问题有帮助
- **为什么不用**: 增加 Token 消耗；简单问题不需要
- **赛题适用性**: 🟡 **可选** — 仅用于高复杂性题目

## 3.5 检索结果融合

### 3.5.1 Reciprocal Rank Fusion (RRF) ⭐
- **是什么**: 基于排名的融合方法：score = Σ 1/(k+rank)，k 通常取 60
- **为什么用**: 不需要分数归一化（不同索引的分数量纲不同）；简单、有效、业界标准；对排名敏感对分数值不敏感——这恰好匹配多路检索的需求
- **为什么不用**: 如果两路的结果高度重叠，RRF 不如加权和方法好
- **赛题适用性**: ✅ **推荐** — 多路检索融合的默认方案

### 3.5.2 加权和融合 (Weighted Sum / CombSUM)
- **是什么**: 对各路检索的分数做归一化后加权求和
- **为什么用**: 可以精细控制各路的权重（条款匹配权重 > 全文匹配）
- **为什么不用**: 需要良好的归一化（各路的分数值域不同）
- **赛题适用性**: 🟡 **可选** — 如果所有路使用相同的评分量纲

### 3.5.3 CombMNZ
- **是什么**: CombSUM 的变体：一个文档出现在更多路中 → 额外加权（乘以路数）
- **为什么用**: 奖励被多路同时检索到的文档（更可能是真相关）
- **为什么不用**: 可能过度奖励容易被检索的常见文档
- **赛题适用性**: 🟡 **可选** — 与 RRF 搭配使用

### 3.5.4 Borda Count
- **是什么**: 每个路对文档投票：第一名得 N-1 分，最后一名得 0 分
- **为什么用**: 公平的排序融合方法；对异常分数鲁棒
- **为什么不用**: 与 RRF 效果接近，RRF 更简单
- **赛题适用性**: 🟡 **可选**

### 3.5.5 级联过滤 (Cascade Filtering)
- **是什么**: 不是"融合"而是"逐级过滤"：实体匹配 → 缩小范围 → BM25 → 再缩小 → 数值过滤 → 再缩小
- **为什么用**: 先精确后模糊，效率最高；每级可设置不同的保留数量
- **为什么不用**: 需要良好的过滤逻辑（如果第一级就过滤掉了正确答案…）
- **赛题适用性**: ✅ **推荐** — 精确匹配优先，模糊检索兜底

## 3.6 特殊检索模式

### 3.6.1 精确条款定位 (Clause Lookup)
- **是什么**: 题目中如果提到了"第47条"或"6.1"，直接在 clause_index 哈希表中 O(1) 查找
- **为什么用**: 零 Token、零延迟、100% 精确
- **为什么不用**: 仅适用于题目中出现条款编号的情况
- **赛题适用性**: ✅ **推荐** — 不必多言的必备功能

### 3.6.2 实体精确匹配 (Entity Lookup)
- **是什么**: 题目中的产品名/公司名/法规名 → Aho-Corasick 多模式匹配 → 直接定位相关 chunks
- **为什么用**: 零 Token、零延迟、100% 精确
- **为什么不用**: 实体提取可能不完整（需要同义词表）
- **赛题适用性**: ✅ **推荐** — 必备

### 3.6.3 数值范围过滤 (Numeric Range)
- **是什么**: 题目中涉及"100万以上" → 在 numeric_index 中筛选所有 >= 1000000 的数值所在 chunks
- **为什么用**: 为计算题缩小搜索范围；精确过滤无关文档
- **为什么不用**: 需要预先提取所有数值（但离线可以做到）
- **赛题适用性**: ✅ **推荐** — 计算题的重要辅助

### 3.6.4 布尔与短语查询 (Boolean & Phrase)
- **是什么**: BM25 的查询语法扩展：`"身故保险金" AND (90万 OR 100万)` 这种精确短语+布尔组合
- **为什么用**: 对结构化查询非常有效；BM25 天然支持短语查询
- **为什么不用**: 需要查询解析（但可以用 Qwen 生成结构化查询）
- **赛题适用性**: 🟡 **可选** — 作为 Qwen 查询输出的格式规范

### 3.6.5 Proximity Search / MRF (Markov Random Field)
- **是什么**: 要求查询词在文档中出现在一定距离内（"身故保险金" 和 "90万" 在 20 个词以内）
- **为什么用**: 精确匹配同时出现的上下文——条款中"身故保险金"和"90万"相邻出现说明高度相关；Huly et al. (2024) 实验证实 MRF 超越 BM25
- **为什么不用**: 实现复杂度高于普通 BM25；需要额外的索引结构
- **赛题适用性**: 🟡 **可选** — V2/V3 增强

### 3.6.6 同义词扩展 (Synonym Expansion)
- **是什么**: 在查询/索引中添加强术语的同义词（"身故"↔"死亡"↔"去世"）
- **为什么用**: 不同文档可能用不同措辞描述同一概念
- **为什么不用**: 金融文档的措辞非常精确——"身故保险金"和"死亡保险金"可能是不同的法律概念（一个条款用"身故"，另一个用"死亡"，区别是有意的）
- **赛题适用性**: ⚠️ **慎用** — 金融/法律术语的同义词扩展需极度谨慎，错误扩展可能降低精度

# Stage 4: 重排序

> **核心问题**: 检索返回 30-50 个候选 → 需要精排到 5-8 个最相关的 → 送给推理模块
> **约束**: 只能用 Qwen 做重排序（不能用 Cross-Encoder/BGE-Reranker 等 embedding 模型）

## 4.1 LLM 重排序方法

### 4.1.1 Pointwise Qwen Rerank ⭐
- **是什么**: 对每个候选文档，让 Qwen 输出 0-10 的相关性分数
- **为什么用**: 简单直接；可以独立判断每个候选（没有候选间干扰）；ERank (Alibaba 2025) 证实 pointwise + RL 训练可追平 listwise
- **为什么不用**: 对 30 个候选逐个判断需要 30 次 Qwen 调用（Token 消耗大）；缺乏候选间比较的全局视角
- **赛题适用性**: 🟡 **可选** — 候选数少时（<10）可用

### 4.1.2 Listwise Qwen Rerank ⭐
- **是什么**: 将所有候选放在一个 prompt 中，让 Qwen 直接排序
- **为什么用**: 一次调用完成全部排序（省 Token）；Qwen 可以做候选间的比较（"A 比 B 更相关"）；RankGPT 范式验证有效
- **为什么不用**: 候选太多时 prompt 超长（30 个候选 × 500 token/候选 = 15,000 token）；U 形注意力导致中间候选被忽略
- **赛题适用性**: ✅ **推荐** — 候选 10-20 个时使用；超过 20 个时分批

### 4.1.3 滑动窗口 Listwise (Sliding Window)
- **是什么**: 将 30 个候选分成多个窗口（每组 10 个），每组内部 listwise 排序，取各组 Top-N 进入下一轮
- **为什么用**: 解决 "30 个候选太多" 的问题；保留 listwise 的全局比较能力
- **为什么不用**: 多轮调用增加 Token；窗口边界处可能有信息丢失
- **赛题适用性**: 🟡 **可选** — 候选超过 20 个时使用

### 4.1.4 Pairwise Qwen Rerank
- **是什么**: 让 Qwen 两两比较："A 和 B 哪个更相关？"
- **为什么用**: LLM 在两两比较时最准确（比 pointwise 判断更一致）
- **为什么不用**: O(n²) 比较次数 → Token 消耗极大（30 个候选 = 435 对比较）；不实用
- **赛题适用性**: ❌ **不可用** — Token 消耗太大

### 4.1.5 生成式重排序 (Generative Rerank)
- **是什么**: 让 Qwen 直接基于检索结果生成答案，不显示排序分数——Qwen 在生成答案时自然"关注"最相关的内容
- **为什么用**: 绕过排序步骤，直接端到端；Qwen 通过注意力机制隐式做软选择
- **为什么不用**: 上下文太长的 U 形问题；缺乏显式的选择可解释性
- **赛题适用性**: 🟡 **可选** — 结合压缩使用（先压缩到 2000 token 内再端到端）

### 4.1.6 ERank 风格重排序
- **是什么**: Qwen 对每个候选输出 0-10 整数分数 + 简短理由，然后按分数排序
- **为什么用**: 精细粒度评分（0-10 vs 二分类）；附带理由增强可解释性
- **为什么不用**: 每个候选需要独立的 Qwen 调用 → Token 消耗较高
- **赛题适用性**: 🟡 **可选** — 候选数少时的精细方案

## 4.2 非 LLM 重排序（零 Token）

### 4.2.1 BM25 分数二次精排
- **是什么**: 对候选的 BM25 分数做二次处理（如结合位置权重、条款权重等）重新排序
- **为什么用**: 零 Token；快速的规则增强
- **为什么不用**: 没有引入新信息（还是那个 BM25）；精度提升有限
- **赛题适用性**: 🟡 **可选** — 作为 LLM 重排序前的粗排

### 4.2.2 规则驱动的重排序
- **是什么**: 用领域规则调整排序权重——同一文档的连续 chunks 提升排名；条款编号精确匹配加权；数值精确匹配加权
- **为什么用**: 零 Token；可解释；对结构化查询效果显著
- **为什么不用**: 规则覆盖不全；对模糊查询无效
- **赛题适用性**: ✅ **推荐** — 所有方案默认的零成本增强

# Stage 5: 上下文压缩

> **核心问题**: 检索到的文本太多（3,000-8,000 token），直接给 Qwen 做推理会超出注意力有效范围且 Token 开销大 → 需要压缩到 500-1,000 token 的高密度信息
> **约束更新**: 默认不使用非 Qwen 小模型压缩。压缩优先采用规则抽取、字段过滤、表格结构化和证据预算裁剪；若使用 Qwen 摘要或自检，必须记录 Token。

## 5.1 抽取式压缩（规则/词法，零 Token 消耗）⭐

### 5.1.0 规则抽取式压缩（默认首选）
- **是什么**: 对检索候选块按题干词、选项词、标题、条款号、表格名、数值、日期、比例、否定词和强义务词打分，保留原文句子/表格行
- **为什么用**: 不依赖任何额外模型；完全保留原文证据；可解释、可复现；对金融数值和法条最稳
- **为什么不用**: 召回质量依赖规则和分块质量；对隐含语义、同义改写不如模型压缩
- **赛题适用性**: ✅ **推荐** — 默认进入提交链路

### 5.1.1 LLMLingua-2（实验项，不作为默认）
- **是什么**: 微软开源——用 BERT 级小模型做 token 级二分类（保留/丢弃），训练数据蒸馏自 GPT-4
- **为什么用**: 毫秒级压缩（比 LLMLingua v1 快 3-6×）；不需要 GPU（CPU 可运行）；保留率可控（如 40%）
- **为什么不用**: BERT 级模型仍是非 Qwen 模型；虽然不是 embedding，但代码审核存在规则解释风险
- **赛题适用性**: ⚠️ **慎用** — 只做本地消融，不进入默认提交链路

### 5.1.2 LongLLMLingua
- **是什么**: LLMLingua 的长上下文版本——增加了文档重排序（把关键文档放开头/结尾）、问题感知过滤
- **为什么用**: 针对 U 形注意力的位置优化（Lost in the Middle 缓解）；问题感知过滤确保与当前问题相关的句子优先保留
- **为什么不用**: 仍依赖非 Qwen 小模型/模型信号；比规则抽取更难解释和复现
- **赛题适用性**: ⚠️ **慎用** — 不作为默认

### 5.1.3 LLMLingua (v1)
- **是什么**: 用 GPT-2-small 或 LLaMA-7B 计算每个 token 的 perplexity，高 perplexity 的 token 更重要（信息量大），按重要性秩保留
- **为什么用**: 理论基础扎实（perplexity ≈ 信息量）；被大量工作验证
- **为什么不用**: 需要运行非 Qwen 小 LM，规则风险高；速度慢于 LLMLingua-2
- **赛题适用性**: ⚠️ **慎用** — 只保留理论参考

### 5.1.4 Selective Context
- **是什么**: 用自我信息量 (self-information) + SpaCy 句法分析来保留句子
- **为什么用**: 完全无参数（不需要任何 LM）；基于句法边界保留完整句子
- **为什么不用**: 中文句法分析精度有限；不如 LLMLingua 系列有效
- **赛题适用性**: 🟡 **可选** — 无模型依赖时使用

### 5.1.5 DSPC (Dual-Stage Progressive Compression, 2025)
- **是什么**: 三信号融合（注意力 + 损失差 + 位置）的渐进式压缩
- **为什么用**: 2025 最新，宣称 SOTA；不需要训练；三信号互补
- **为什么不用**: 需要访问 LLM 内部注意力（要用 Qwen API 的话拿不到）；可以先用 Qwen 转成小模型信号
- **赛题适用性**: ⚠️ **慎用** — 需要 LLM 内部信号，与 API-only 模式不兼容

### 5.1.6 CPC (Context-aware Prompt Compression, AAAI 2025)
- **是什么**: 对比学习训练的句子级编码器，预测句子对当前问题的相关性
- **为什么用**: 比 LLMLingua 快 11×；句子级保留（比 token 级更语义完整）
- **为什么不用**: 需要加载对比学习句子编码器（一个小模型）；中文支持需验证
- **赛题适用性**: ⚠️ **慎用** — 思路可借鉴，不加载模型

### 5.1.7 FrugalPrompt
- **是什么**: 2025 年新工作——基于 token 预算约束的最优句子选择
- **为什么用**: 明确针对 Token 预算优化；自动平衡覆盖度和 Token 数
- **为什么不用**: 较新，社区验证较少
- **赛题适用性**: 🟡 **可选** — 实现为规则句子选择时可用

## 5.2 摘要式压缩（用 Qwen，计 Token）

### 5.2.1 Qwen 摘要压缩
- **是什么**: 让 Qwen 将检索文本压缩为几百字的摘要
- **为什么用**: 最语义化的压缩；可以跨文档整合信息
- **为什么不用**: 消耗 Qwen Token（~500-2000 token）；可能丢失精确数值（"120亿元"可能被总结为"约120亿元"）——这对金融计算题是致命的
- **赛题适用性**: ⚠️ **慎用** — 仅适用于不需要精确数值的推理判断题；计算题坚决不用

### 5.2.2 Map-Reduce 摘要
- **是什么**: 每篇文档独立摘要（Map）→ 合并所有摘要 → 再摘要（Reduce）
- **为什么用**: 处理超长文档的标准方案；每步的上下文可控
- **为什么不用**: 每次摘要可能丢失信息，多次摘要累积误差；Token 消耗大；精确数值丢失风险高
- **赛题适用性**: ❌ **不可用** — 信息损失不可控，不适合金融精确问答

### 5.2.3 逐段精炼 (Refine)
- **是什么**: 顺序处理每段文本，基于已有摘要和新文本逐步精炼摘要
- **为什么用**: 比 Map-Reduce 丢失信息少
- **为什么不用**: 串行处理（不能并行）→ 延迟高；仍然有精确数值丢失风险
- **赛题适用性**: ⚠️ **慎用**

## 5.3 软提示压缩（需要 LLM 内部访问 → 不可用）

### 5.3.1 AutoCompressor (Gisting)
- **是什么**: 训练 LLM 将长上下文压缩为少量特殊"gist token"，后续用这些 token 代替全文
- **为什么用**: 压缩比极高（20×+）；软 token 能编码超过明文的信息密度
- **为什么不用**: **需要训练和修改 LLM → 违反禁止微调规则！** 🚫 且需要 LLM 内部访问（Qwen API 不支持）
- **赛题适用性**: ❌ **不可用**

### 5.3.2 ICAE (In-context Autoencoder)
- **是什么**: 用编码器-解码器将上下文压缩为少量 memory token
- **为什么用**: 同样压缩比极高
- **为什么不用**: **需要额外模型（编码器）→ 违反仅限 Qwen 的规则！** 🚫
- **赛题适用性**: ❌ **不可用**

### 5.3.3 500xCompressor / xRAG
- **是什么**: 利用 KV Cache 值实现 480× 压缩
- **为什么用**: 最极端的压缩比
- **为什么不用**: **需要 LLM 内部 KV Cache 访问 → Qwen API 不支持！** 🚫 且需要额外模型
- **赛题适用性**: ❌ **不可用**

> **软提示压缩小结**: 这整条分支都是不可用的——它们都需要 LLM 内部访问（KV cache / 梯度 / 参数修改）或额外模型。本赛题只能用 Qwen API（黑盒），所以软提示技术无从施展。

## 5.4 结构化压缩

### 5.4.1 FactCard (预生成结构化摘要) ⭐
- **是什么**: 离线阶段对每篇 PDF 生成 150-200 字的结构化摘要（产品名、关键条款编号+内容、关键数值、日期、计算公式）
- **为什么用**: 一次生成，100 题复用（Token 分摊到全部题目）；快速跨文档比较
- **为什么不用**: 如果用 Qwen 生成，必须计入 Token；摘要可能丢失精确数字或例外条件
- **赛题适用性**: 🟡 **可选** — 推荐优先用规则/KVP生成；Qwen 生成版本只在有完整 Token 统计时使用

### 5.4.2 关键信息抽取 (KVP extraction)
- **是什么**: 离线抽取每篇文档的 Key-Value Pairs（产品名→条款内容、数值范围、计算公式）
- **为什么用**: 结构化信息可以直接做精确匹配和计算；不依赖词法检索
- **为什么不用**: 离线提取不完整可能导致遗漏；需要领域特定的提取模板
- **赛题适用性**: ✅ **推荐** — 保险和财报领域的关键信息抽取

### 5.4.3 表格 → JSON 结构化
- **是什么**: 财报表格→结构化 JSON，保留行列关系
- **为什么用**: 表格在 JSON 中比在纯文本中更易于查询和计算
- **为什么不用**: 复杂表格（嵌套表头、合并单元格）的结构化可能出错
- **赛题适用性**: ✅ **推荐** — financial_reports 必备

## 5.5 保真度保护

### 5.5.1 正则保护规则 ⭐
- **是什么**: 在压缩前用正则标记关键行（金额/百分比/日期/条款编号/产品名），压缩时无条件保留
- **为什么用**: 压缩的最大风险是丢失精确数值——金融 QA 中 60%+ 的错误由此引起；零成本
- **为什么不用**: 没有理由不用
- **赛题适用性**: ✅ **推荐** — 默认内置

### 5.5.2 数值验证
- **是什么**: 压缩后检查：原始文本中的关键数值在压缩文本中是否仍然存在？
- **为什么用**: 事后验证，防止重要数值在压缩中丢失
- **为什么不用**: 需要额外的验证步骤（但离线/免费）
- **赛题适用性**: ✅ **推荐** — 零成本保险

# Stage 6: 记忆管理

> **核心问题**: 多文档、多轮推理过程中，如何管理已获得的信息？哪些证据需要保留？哪些可以遗忘？如何组织累积的证据？
> **关键公式**: 记忆六操作 = Consolidation + Updating + Indexing + Forgetting + Retrieval + Compression

## 6.1 记忆架构

### 6.1.1 扁平记忆 (Flat / All-in-Prompt)
- **是什么**: 所有检索到的证据直接拼进 Prompt，一次提交
- **为什么用**: 最简单；不需要记忆管理机制
- **为什么不用**: Token 消耗与证据量线性增长；U 形注意力导致中间证据利用率低；容易上下文污染
- **赛题适用性**: ⚠️ **慎用** — 仅简单题可用

### 6.1.2 结构化证据链 ⭐
- **是什么**: 每个选项的证据独立维护：[选项A的证据] [选项B的证据] ... 选项间互不干扰
- **为什么用**: 自然支持逐选项独立判断；避免上下文干扰；可追踪每个选项的证据来源
- **为什么不用**: 需要维护多个独立的证据缓冲区
- **赛题适用性**: ✅ **推荐** — 多选/单选题的核心记忆组织方式

### 6.1.3 层级化记忆 (Hierarchical)
- **是什么**: L1(当前选项证据) → L2(当前题目已判断选项) → L3(同文档已完成题目) → L4(领域背景知识)
- **为什么用**: 不同层级的信息存活周期不同；L2 的同题证据在全部选项判断后可释放；L3 的同文档信息可复用
- **为什么不用**: 实现复杂度较高；需要设计清理策略
- **赛题适用性**: ✅ **推荐** — V3 的批处理架构

### 6.1.4 MemGPT 风格记忆
- **是什么**: LLM 主动管理自己的记忆——通过函数调用读写记忆存储
- **为什么用**: 记忆操作由 LLM 自主决策（而非固定规则）；灵活性强
- **为什么不用**: 需要多次 LLM 调用（记忆读写都要消耗 Token）；记忆质量依赖 LLM 的判断
- **赛题适用性**: 🟡 **可选** — V3 实验项

### 6.1.5 MemWalker 树状记忆
- **是什么**: 将长文本构建为树状结构，LLM 沿树路径导航收集信息
- **为什么用**: 适合需要导航式阅读的场景
- **为什么不用**: 导航过程消耗大量 Token；导航路径可能错过关键信息
- **赛题适用性**: ⚠️ **慎用** — Token 开销大，不如直接检索

### 6.1.6 KV Cache 复用记忆
- **是什么**: 利用 Qwen API 的 prompt caching 功能，重复的 context 前缀自动缓存
- **为什么用**: 如果 Qwen API 支持 prompt caching（阿里云百炼可能有），同文档题目的公共前缀可复用，节省 Token
- **为什么不用**: 依赖 API 实现（需确认阿里云百炼是否支持）；仅对前缀匹配有效
- **赛题适用性**: 🟡 **可选** — 如果 API 支持，直接受益

### 6.1.7 外部记忆数据库
- **是什么**: 用 SQLite/JSON 存储已获得的证据 → 后续题目可查询
- **为什么用**: 同文档题目间知识复用；跨题目的证据不会丢失
- **为什么不用**: 增加系统复杂度；需要设计查询逻辑
- **赛题适用性**: 🟡 **可选** — 批处理方案的辅助

## 6.2 记忆操作

### 6.2.1 Consolidation (合并)
- **是什么**: 同一题目的多个证据片段合并为一个统一的证据视图
- **为什么用**: 消除重复信息；提供完整上下文给推理模块
- **为什么不用**: 纯合并操作——无代价
- **赛题适用性**: ✅ **推荐**

### 6.2.2 Indexing (索引)
- **是什么**: 给每条证据打标签（关联哪个选项、哪个文档、哪个条款）
- **为什么用**: 快速按标签检索证据
- **为什么不用**: 需要维护标签系统
- **赛题适用性**: ✅ **推荐**

### 6.2.3 Forgetting (遗忘)
- **是什么**: 已判断完成的选项证据释放；已完成的题目记忆清空；无关信息清理
- **为什么用**: 防止上下文污染（已完成选项的证据干扰后续选项）；释放 Token 空间
- **为什么不用**: 需要确保遗忘的是真无关信息（不能误删）
- **赛题适用性**: ✅ **推荐** — 逐选项判断的关键步骤

### 6.2.4 Updating (更新)
- **是什么**: 更强的证据替换弱的证据；新信息覆盖旧信息
- **为什么用**: IterKey 迭代检索时，新检索结果可能优于旧结果
- **为什么不用**: 只影响迭代检索场景
- **赛题适用性**: 🟡 **可选**

### 6.2.5 Compression (上下文内压缩)
- **是什么**: 将多段证据压缩为"关键事实摘要"放入 Prompt
- **为什么用**: 减少 Prompt 长度
- **为什么不用**: 压缩可能丢失细节（见 Stage 5）
- **赛题适用性**: 🟡 **可选**

### 6.2.6 Retrieval (记忆检索)
- **是什么**: 从已存储的记忆中按关键词/条件检索相关信息
- **为什么用**: 避免重复读 PDF（跨题目复用）
- **为什么不用**: 需要维护记忆索引
- **赛题适用性**: 🟡 **可选**

# Stage 7: 推理与提示工程

> **核心问题**: 证据已就位，如何让 Qwen 给出正确答案？
> **最大 Token 开销所在**

## 7.1 基本推理策略

### 7.1.1 直接回答 (Direct Answer)
- **是什么**: 证据 + 题干 + 选项 → Qwen 直接输出答案
- **为什么用**: 零推理设计成本；最快
- **为什么不用**: 对复杂推理题（计算、多条件判断）准确率低
- **赛题适用性**: ⚠️ **慎用** — 仅基线

### 7.1.2 链式思维 (Chain-of-Thought, CoT) ⭐
- **是什么**: 提示 Qwen "让我们一步步分析"，输出推理链后再给出答案
- **为什么用**: 最核心的推理增强——迫使 LLM 在输出答案前经历完整推理过程；对计算和逻辑推理题效果显著
- **为什么不用**: 增加输出 Token（推理链通常 200-500 token）；可能生成错误的推理链（错得更自信）
- **赛题适用性**: ✅ **推荐** — 所有题型的默认推理策略

### 7.1.3 逐选项独立 CoT ⭐
- **是什么**: 对多选的每个选项独立执行 CoT 推理（4 个独立调用），最后汇总
- **为什么用**: 消除选项间干扰（上下文污染）；每个选项获得独立的推理空间；汇总时只需检查"哪些选项成立"
- **为什么不用**: 4 个独立调用 = 4 倍推理 Token（但总比把 4 个选项全放一起让 LLM 混乱要好）
- **赛题适用性**: ✅ **推荐** — 多选题的核心策略（多选占一半以上！）

### 7.1.4 排除法推理 (Elimination Reasoning)
- **是什么**: 逐一排除明显错误的选项，在剩余选项中 CoT 选择
- **为什么用**: 人类做多选题的自然策略；排除简单错误后，剩余选项的比较更精细
- **为什么不用**: ACL 2024 研究显示 LLM 不擅长"推理到错误"（PoE 总弱于直接选择正确）
- **赛题适用性**: 🟡 **可选** — 可结合使用但不建议作为主要策略；逐选项独立判断更好

### 7.1.5 ETS (Eliminate-Then-Select) 框架
- **是什么**: 2025 新框架——每选项独立 CoT + 置信度 → 基于置信度过滤 → 剩余选项精细比较
- **为什么用**: 准确率 +4.3%（6 个 QA 基准平均）；结合了独立判断和全局比较的优势
- **为什么不用**: Token 消耗更多（每个选项 CoT + 最后还有全局比较）
- **赛题适用性**: 🟡 **可选** — V3 低置信题使用

### 7.1.6 ARR (Analyze → Retrieve → Reason)
- **是什么**: 分解为分析意图 → 检索信息 → 逐步推理
- **为什么用**: 结构化推理步骤；"分析意图"步骤显著提升效果
- **为什么不用**: 增加推理步骤 → Token 更多
- **赛题适用性**: 🟡 **可选** — 高难度题目使用

### 7.1.7 角色提示 (Role Prompting)
- **是什么**: "你是一名资深的保险精算师/证券律师/注册会计师..."
- **为什么用**: NAACL 2025 研究显示角色提示在 80% 的 pointwise 任务中有效；激发领域知识
- **为什么不用**: 对某些推理任务无效甚至有害（角色可能限制模型的推理广度）
- **赛题适用性**: ✅ **推荐** — 作为提示模板的一部分，成本极低

### 7.1.8 Few-Shot 示例
- **是什么**: 在 Prompt 中给出 3-5 个同类型问答示例（问题→推理→答案）
- **为什么用**: 显著提升复杂任务的稳定性；Qwen 对示例学习敏感
- **为什么不用**: 每个示例增加 200-500 Token 的 Prompt 开销；但分摊到所有题目就值了
- **赛题适用性**: ✅ **推荐** — 每个领域准备 3-5 个示例

### 7.1.9 领域特化 Prompt 模板
- **是什么**: 不同领域的 Prompt 模板不同——保险（强调计算公式+条款触发条件）、法规（强调法条编号+适用条件）、财报（强调数值精确+跨年对比）
- **为什么用**: 不同领域的推理逻辑差异大；领域特化 Prompt 提供最相关的指导
- **为什么不用**: 维护 5 套模板
- **赛题适用性**: ✅ **推荐** — 少量 Token 投入，显著效果提升

## 7.2 高级推理策略

### 7.2.1 Calculator Agent ⭐
- **是什么**: Qwen CoT 推理 → 提取计算公式和代入数值 → Python 执行精确计算 → 返回结果
- **为什么用**: LLM 心算错误率极高（特别是百万级金额的精确比较）；Python 计算 100% 准确；消除 88% 算术错误
- **为什么不用**: 需要实现安全的 Python 执行沙箱；公式提取可能出错（Qwen 输出了错误的公式）
- **赛题适用性**: ✅ **推荐** — 保险和财报计算题的必备组件

### 7.2.2 Self-Consistency (SC)
- **是什么**: 同一问题独立推理 3-5 次 → 多数投票选出最终答案
- **为什么用**: 显著提升准确率（采样多次，答案分布反映置信度）；数学/逻辑推理题效果最好
- **为什么不用**: Token 消耗倍数增长（3-5 倍推理 Token）；对简单事实查询效果有限（3 次答案一致，投票无增益）
- **赛题适用性**: 🟡 **可选** — V3 使用，仅用于高难度/低置信题目

### 7.2.3 Chain-of-Verification (CoVe)
- **是什么**: 生成答案后，让 Qwen 反查证据验证自己的答案是否正确
- **为什么用**: 捕获幻觉和推理错误；少量额外 Token（~1000 token）
- **为什么不用**: 如果证据本身不全，验证也无法纠正
- **赛题适用性**: ✅ **推荐** — V2/V3 的低置信验证步骤

### 7.2.4 二次确认 (Double-Check)
- **是什么**: 置信度 < 阈值 → 触发完整重推理（可能用更多证据/不同推理路径）
- **为什么用**: 对低置信度答案做额外的安全保障
- **为什么不用**: 需要可靠的置信度估计（LLM 的置信度往往不准——2025 研究显示推理会让 LLM 对错误答案更自信）
- **赛题适用性**: 🟡 **可选** — V3 使用

### 7.2.5 反向验证 (Reverse Verification)
- **是什么**: 假设答案 X 是正确的 → 验证这个假设是否与所有证据一致 → 如果不一致则排除
- **为什么用**: 逆向思维可能发现正向推理遗漏的矛盾
- **为什么不用**: 增加推理 Token；简单题不需要
- **赛题适用性**: 🟡 **可选** — 高难度多选判断

### 7.2.6 Program-of-Thought (PoT)
- **是什么**: CoT → 生成 Python 程序代码 → 执行代码得到计算结果
- **为什么用**: 对于复杂的多步计算（涉及多个公式、多个产品对比），Python 比 LLM 心算可靠得多
- **为什么不用**: 比 Calculator Agent 更复杂（需要生成和执行完整程序）；Qwen 生成的代码可能语法错误
- **赛题适用性**: 🟡 **可选** — Calculator Agent 的高级版，适用于极复杂计算

### 7.2.7 Contrastive CoT (C-CoT)
- **是什么**: 同时推理"A 为什么对"和"A 为什么错"，综合判断
- **为什么用**: 避免确认偏误（只看到支持 A 成立的证据）
- **为什么不用**: 推理 Token 翻倍
- **赛题适用性**: 🟡 **可选** — 高难度选项判断

### 7.2.8 Decomposition (最少到最多)
- **是什么**: 将复杂问题分解为子问题，逐个解决，最后合并
- **为什么用**: 多步推理的标准策略
- **为什么不用**: 子问题分解和合并都需要额外的 LLM 调用
- **赛题适用性**: 🟡 **可选** — 复杂多文档比较题

### 7.2.9 Step-Back Prompting
- **是什么**: 先问抽象/原理性问题（"身故保险金的计算遵循什么规则？"），再问具体问题（"各产品具体是多少？"）
- **为什么用**: 对不知道从哪开始的问题有帮助
- **为什么不用**: Token 消耗加倍
- **赛题适用性**: ⚠️ **慎用** — 大多数赛题已经足够具体

### 7.2.10 Tree-of-Thought (ToT) / Graph-of-Thought
- **是什么**: 探索多条推理路径，在树/图结构中回溯和选择
- **为什么用**: 最全面的推理探索；理论上能发现 CoT 遗漏的正确路径
- **为什么不用**: Token 消耗极大（每步推理都可能分叉）；对大部分赛题过度设计；非必要不使用
- **赛题适用性**: ❌ **不可用** — Token 消耗过大，仅对极少数超难题有价值

# Stage 8: 答案处理与验证

> **核心问题**: 从 Qwen 的文本输出中提取标准格式答案，验证合法性
> **约束**: 答案格式必须严格匹配（mcq: "A", multi: "ABC", tf: "A"/"B"）

## 8.1 答案提取

### 8.1.1 正则提取
- **是什么**: re.findall(r'[A-D]', output.upper())
- **为什么用**: 简单、快速、可靠
- **为什么不用**: 如果 Qwen 输出"答案:D"→ 提取 A 和 D 而不是只有 D
- **赛题适用性**: ✅ **推荐** — 配合更严格的解析使用

### 8.1.2 结构化输出提示
- **是什么**: 要求 Qwen 以特定格式输出：`ANSWER: A`
- **为什么用**: 比自由文本容易解析得多；减少提取错误
- **为什么不用**: 强硬格式可能限制推理质量（Qwen 可能在思考阶段需要自由文本）
- **赛题适用性**: ✅ **推荐** — 推理阶段自由文本，最后一行结构化

### 8.1.3 JSON 结构化输出
- **是什么**: 要求 Qwen 输出 JSON：`{"answer": "AC", "reasoning": "...", "evidence": [...]}`
- **为什么用**: 最精确的解析；附带推理和证据便于调试
- **为什么不用**: Qwen 可能不严格遵守 JSON 格式（需要 try/except 兜底）；JSON 输出增加 Token
- **赛题适用性**: ✅ **推荐** — Qwen3.6 的 JSON 输出能力较强

## 8.2 答案验证

### 8.2.1 格式校验
- **是什么**: 检查答案是否符合题型格式（mcq: 单个 A-D, multi: 排序的多个 A-D, tf: A/B）
- **为什么用**: 防止因格式错误失分——不合法的答案直接计 0 分
- **为什么不用**: 没有理由不用
- **赛题适用性**: ✅ **推荐** — 最后一道安全防线

### 8.2.2 去重 + 排序
- **是什么**: 多选题：收集所有"成立"选项 → sorted(set(options)) → join
- **为什么用**: 赛题要求多选题按字母排序，不排序直接计错误
- **为什么不用**: 没有理由不用
- **赛题适用性**: ✅ **推荐** — 必选

### 8.2.3 非法字符清理
- **是什么**: 移除空格、逗号、小写字母等非法字符
- **为什么用**: 防止格式错误（"A, C" → "AC"）
- **为什么不用**: 需确保清理不会误解合法内容
- **赛题适用性**: ✅ **推荐**

### 8.2.4 答案合理性检查
- **是什么**: 规则检查：单选的答案必然是单个字母；多选的答案至少有 1 个字母；全选（ABCD）可能有问题需复查
- **为什么用**: 识别明显错误的答案模式
- **为什么不用**: 规则检查不能发现推理错误
- **赛题适用性**: ✅ **推荐** — 零成本附加保障

# Stage 9: Token 优化策略

> **核心公式**: FinalScore = 100 × Accuracy × (0.7 + 0.3 × TokenScore)
> **TokenScore**: max(0, min(1, (5,000,000 - TotalTokens) / 5,000,000))
> **70% 准确率权重，30% Token 效率权重**。Token 少用 100 万 → TokenScore 从 0.8→1.0，得分提升约 6%

## 9.1 批处理策略

### 9.1.1 同文档题目分组批处理 ⭐
- **是什么**: 将 A 榜 100 题按涉及的文档分组。同文档的多道题目按顺序处理——第一题检索的证据可供后续题目复用
- **为什么用**: 避免重复检索和重复阅读同一篇文档；DocProfile/KVP 和候选证据复用；节省 20-40% Token
- **为什么不用**: 需要额外的题目分组逻辑；不同题目可能需要不同的检索策略
- **赛题适用性**: ✅ **推荐** — V3 的核心设计

### 9.1.2 共享上下文前缀
- **是什么**: 将同文档多题的"公共上下文"（规则画像、KVP、共享证据）放在 Prompt 前缀，各题只需追加差异化内容
- **为什么用**: 如果阿里云百炼 API 支持 prompt caching，前缀部分只需一次编码
- **为什么不用**: 依赖 API 支持
- **赛题适用性**: 🟡 **可选** — 取决于 API 能力

### 9.1.3 先易后难路由 ⭐
- **是什么**: 先评估题目难度 → 简单题用轻量策略（省 Token）→ 难题用完整策略（保准确率）
- **为什么用**: 不是所有题都需要全套推理——简单事实查询可以极简处理；30-40% 的 Token 可以省在简单题上
- **为什么不用**: 需要准确的难度评估机制
- **赛题适用性**: ✅ **推荐** — V3 的核心设计

### 9.1.4 级联策略升级
- **是什么**: 先用 V1（省 Token 模式）→ 置信度低 → 升级到 V2 → 仍低置信 → 升级到 V3
- **为什么用**: 只为难题花 Token；简单题不浪费
- **为什么不用**: 每次升级重做检索+推理，重复消耗（但值得，因为大多数题在第一级就能解决）
- **赛题适用性**: ✅ **推荐** — V3 的核心设计

## 9.2 推理优化

### 9.2.1 推理 Token 预算控制
- **是什么**: 为每题的推理设 Token 上限（如 45,000 token），实时监控，接近上限时降级策略
- **为什么用**: 防止某道难题消耗过多 Token 导致总预算不足
- **为什么不用**: 过于激进的截断可能影响答案质量
- **赛题适用性**: ✅ **推荐** — 所有方案都应内置

### 9.2.2 输出长度限制
- **是什么**: 调用 Qwen 时设置合理的 max_tokens（如推理 500、答案 50）
- **为什么用**: 防止 Qwen 生成冗长的无用输出；completion_tokens 也计入总预算
- **为什么不用**: 过于严格的限制可能截断推理链
- **赛题适用性**: ✅ **推荐** — 精确控制

### 9.2.3 推理路径缓存
- **是什么**: 相似题目（如都是"比较 A 和 B 的退保金额"）复用推理模板和部分检索结果
- **为什么用**: 避免重复推理
- **为什么不用**: 题目间的相似度判断可能不准确
- **赛题适用性**: 🟡 **可选**

### 9.2.4 早停策略 (Early Stopping)
- **是什么**: 在逐选项判断中，如果前 2 个选项已经确定某多选不可能全对/全错，减少后续选项的推理深度
- **为什么用**: 节省不必要的深度推理
- **为什么不用**: 可能导致漏判
- **赛题适用性**: ⚠️ **慎用** — 节省有限但风险不低

## 9.3 检索优化

### 9.3.1 检索次数上限
- **是什么**: 迭代检索最多 2-3 轮，防止无限循环
- **为什么用**: 防止检索陷入死循环
- **为什么不用**: 极少数题可能需要更多轮
- **赛题适用性**: ✅ **推荐**

### 9.3.2 候选数量上限
- **是什么**: 粗筛 Top-50，精筛 Top-15，最终选 Top-5 送推理
- **为什么用**: 每个阶段的候选数严格控制，避免过度消费 Token
- **为什么不用**: 过于激进的裁剪可能丢失关键证据
- **赛题适用性**: ✅ **推荐**

## 9.4 全局策略

### 9.4.1 实时预算监控
- **是什么**: 全局 Token 计数器，在线的每题消耗 vs 预算余额实时展示
- **为什么用**: 避免最后发现预算不够
- **为什么不用**: 没有理由不用
- **赛题适用性**: ✅ **推荐**

### 9.4.2 预算预留
- **是什么**: 为 B 榜预留 50% 预算（约 2,500,000 token），不全部花在 A 榜
- **为什么用**: B 榜更难（盲搜），需要更多 Token
- **为什么不用**: B 榜题目 100 道，与 A 榜同数
- **赛题适用性**: ✅ **推荐** — 全局策略

### 9.4.3 Token 分层预算
- **是什么**: 把 5,000,000 token 预算分配到每个领域（按题目数+复杂度加权）
- **为什么用**: 确保每个领域都有足够的 Token
- **为什么不用**: 领域间差异可能很大
- **赛题适用性**: 🟡 **可选**
