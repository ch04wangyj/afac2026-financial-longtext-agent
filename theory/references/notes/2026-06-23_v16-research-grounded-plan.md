# V16 改进计划：基于论文调研的算法重构

> 制定日期：2026-06-23
> 基线：V14 官网得分 68.69，Token 312,541，反推准确率约 70/100
> V15 已失败：66.7267，人工核验不可靠
> 目标：85+ 分（准确率 87/100）

## 一、论文调研发现

### 1.1 FinAgent-RAG（arXiv 2605.05409，2026-05）

**最直接相关的论文。** Agentic RAG 框架，FinQA/ConvFinQA/TAT-QA 分别达 76.81%/78.46%/74.96%。

三大创新可直接借鉴：
- **Program-of-Thought (PoT)**：生成可执行 Python 代码做精确算术，消除 88% 算术错误
- **自适应策略路由器**：简单题用 CoT（便宜），复杂题用 PoT（贵），API 成本降 41.3%
- **自验证迭代检索**：最多 3 轮检索-推理循环，REJECT 时自动 query refine 重检索

### 1.2 MimirRAG（arXiv 2605.25030，2026-05）

FinanceBench 达 89.3%。三大使能因素：
- **表格感知分块**：保留表格结构而非拆成普通文本
- **元数据集成**：每块附带 doc_type/section/page/table_id 等元数据
- **智能体工作流**：查询规划 → 混合检索 → 验证 → 生成

### 1.3 DCRC（arXiv 2605.31064，KDD 2026）

解决数值幻觉。核心：**编译执行推理**——把查询和证据转为可验证可执行程序，而非让 LLM 心算。

### 1.4 Two-Phase Retrieval（arXiv 2605.20684）

解决"相似性-效用差距"。BM25 排名靠前的可能只是"话题相关"而非"决策有用"。第二阶段用 LLM-as-a-Judge 按"分析效用"重新排序。

### 1.5 MARDoc（arXiv 2606.05749，2026-06）

三智能体：Explorer（多粒度检索）→ Refiner（蒸馏结构化证据）→ Reflector（检查充分性+定向反馈）。**结构化记忆**替代不断增长的上下文。

## 二、赛题约束下的可行性筛选

| 论文方法 | 是否可用 | 原因 |
|---|---|---|
| 对比式检索器（FinAgent-RAG） | ❌ | 需要 embedding 训练，赛题禁止 |
| PoT 程序化推理（FinAgent-RAG） | ✅ | 让 Qwen 生成代码，确定性执行，不违反约束 |
| 自验证迭代检索（FinAgent-RAG） | ✅ | Qwen 做 verifier，BM25 做检索 |
| 自适应策略路由（FinAgent-RAG） | ✅ | 按题型路由 thinking/no-thinking |
| 表格感知分块（MimirRAG） | ✅ | V14 已部分实现，需深化 |
| 元数据集成（MimirRAG） | ✅ | 已有 extra_index_fields，需扩展 |
| 编译执行推理（DCRC） | ✅ | 受限 DSL 执行，V11 已有雏形 |

| 结构化记忆（MARDoc） | ✅ | 替代当前的单次上下文 |

## 三、V16 技术方案

### 核心改造：从"单次检索+模型判断"到"迭代检索+程序化推理+自验证"

```
题目输入
    │
    ▼
┌──────────────┐
│ 查询分解器    │ → 子问题 [s1, s2, ...]
└──────┬───────┘
       │
┌──────▼───────┐
│ 自适应路由器  │ → simple: CoT / complex: PoT
└──┬───────┬───┘
   │       │
┌──▼──┐ ┌──▼──┐
│ CoT │ │ PoT │ (确定性执行)
└──┬──┘ └──┬──┘
   └──┬───┘
┌──────▼───────┐
│ 自验证器      │ → ACCEPT / REJECT
│ (3项检查)    │
└──────┬───────┘
  ACCEPT│  REJECT│
       │ ┌──────▼──────┐
       │ │ Query Refine│ → 重检索（最多3轮）
       │ └─────────────┘
       ▼
   最终答案
```

### 改造 1：Program-of-Thought 数值推理（解决算术错误）

**问题**：当前模型靠"心算"比较数值，V15 有 3 题 conf=0 因证据检索到但计算错误。

**方案**：
- 检测题干中的比较谓词（高于/低于/超过/同比/占比）
- 让 Qwen 生成受限计算代码（仅 compare/difference/ratio/growth_rate）
- 确定性执行，不依赖模型心算
- V11 已有 `numeric_fact_ledger`，扩展为程序化推理

**约束**：不修改基座模型，只让 Qwen 生成代码文本，本地确定性执行。

### 改造 2：自验证迭代检索（解决证据不完整）

**问题**：当前单次检索，conf=0 时直接放弃，3 题因此失分。

**方案**（借鉴 FinAgent-RAG）：
- 首轮检索后，Qwen 做 3 项验证：
  1. 证据是否覆盖所有选项
  2. 关键数值是否有原文支撑
  3. 推理链是否完整
- REJECT 时自动 query refine（换同义词、扩大范围、换检索路径）
- 最多 3 轮，每轮排除已检索过的证据
- 结构化记忆（借鉴 MARDoc）：每轮蒸馏证据而非累积原始上下文

**Token 预算**：每轮约 2-3K token，3 轮最多 +9K token/题，100 题 +900K → 需控制
- 只对 conf<0.8 的题启动迭代（预估约 30 题）
- 增量约 +90K token，总量约 400K

### 改造 3：自适应策略路由（优化 Token）

**问题**：当前全部 --no-thinking，复杂多选题推理深度不足。

**方案**（借鉴 FinAgent-RAG）：
- **简单题（mcq + tf，35 题）**：no-thinking，快速低成本
- **复杂题（multi，65 题）**：启用 thinking，深度推理
- **数值比较题**：路由到 PoT，生成代码执行
- 预估 Token：thinking 增加 ~120K，PoT 增加 ~30K，总量约 460K（TokenScore ≥ 0.90）

### 改造 4：LLM-as-a-Judge 效用重排（解决相似性-效用差距）

**问题**：BM25 排名靠前的可能只是"话题相关"而非"答案有用"。

**方案**（借鉴 Two-Phase Retrieval）：
- 第一阶段：BM25 检索 Top-30 候选（现有）
- 第二阶段：Qwen 做 judge，按"对回答此问题的有用性"重新排序
- 只取 Top-5 进入推理上下文
- **不是 embedding reranker**，是 LLM 评分，合规

### 改造 5：全领域 V15 索引覆盖（扩大优化范围）

**问题**：60% 题目（insurance/regulatory/contracts）仍用 V13 旧语料。

**方案**：
- insurance：条款编号原子化，免责条款层级展开
- regulatory：法规条文按施行日期/义务/时限/措施分槽位
- contracts：募集说明书按发行规模/利率/期限/增信结构化
- 纯离线，零在线 Token 成本

## 四、分阶段实施

### 阶段 1：PoT 数值推理 + 全领域索引（3 天）

1. 实现 `agent/reasoning/self_verifier.py`：
   - 3 项验证检查
   - Query refine 策略
   - 结构化证据记忆
2. 集成到 precise_verifier
3. 对 conf<0.8 的题启动迭代

### 阶段 3：自适应路由 + LLM Judge 重排（2 天）

1. 实现策略路由器：按题型/metadata 路由 thinking/no-thinking/PoT
2. 实现 LLM-as-a-Judge 重排模块
3. Token 预算控制

### 阶段 4：全量运行 + 保守融合 + 提交（2 天）

1. 100 题全量运行
2. **不依赖人工核验**——用模型自验证做接受/拒绝
3. 开发集 3/3 门禁
4. 官网提交验证

**若准确率 80/100**：得分 = 80 × (0.7 + 0.3 × 0.914) = **78.0**
**若准确率 85/100**：得分 = 85 × (0.7 + 0.3 × 0.914) = **82.9**
**若准确率 87/100**：得分 = 87 × (0.7 + 0.3 × 0.914) = **84.8** → 接近 85

**关键：Token 增加会降低 TokenScore，必须确保准确率提升足够大。**

## 六、关键约束（不可违反）

- **不依赖人工核验改答案**（V15 教训）
- PoT 只用受限 DSL（compare/difference/ratio/growth_rate），不允许任意代码
- 自验证用 Qwen，不引入非 Qwen 模型
- LLM Judge 用 Qwen，不是 embedding reranker
- 不修改基座模型参数
- 开发集 3/3 硬门禁
- 在线 Token 控制在 500K 以内（TokenScore ≥ 0.90）
