# LogicRAG paper-grounded execution contract (arXiv:2508.06105v2)

Source paper: Shengyuan Chen et al., "You Don’t Need Pre-built Graphs for RAG: Retrieval Augmented Generation with Adaptive Reasoning Structures", arXiv:2508.06105v2.

Purpose of this note: capture what the paper actually specifies operationally, so later AFAC2026 work can distinguish paper-faithful behavior from local adaptation ideas.

## Short contract

LogicRAG is a query-time structured RAG pipeline, not a pre-built graph system. It:
1. uses an LLM to decompose the input query into subproblems,
2. uses the LLM to infer a DAG of logical dependencies among those subproblems,
3. topologically sorts the DAG into ranks / a forward execution order,
4. retrieves and resolves subproblems rank by rank,
5. prunes at two levels: graph pruning by merging same-rank subproblems into one unified query, and context pruning by maintaining a rolling summarized memory,
6. can dynamically append new subproblems if retrieval for a node is insufficient,
7. uses sampling without replacement to avoid repeated near-duplicate subquery generation,
8. treats moderate top-k as the best efficiency/effectiveness trade-off.

Important boundary: Algorithm 1 explicitly uses an embedding retriever `R(q; θ, K)`. The paper's structural contract is query-time DAG + pruning + rolling memory, but its base retrieval stack is still embedding-based, not lexical/BM25.

## 1) How the paper decomposes queries into subproblems

Paper text:
- "LogicRAG begins by decomposing the input query into a set of subproblems and constructing a directed acyclic graph to model the logical dependencies among them."
- In "Graph construction": "The query Q is first segmented into subproblems via LLM-based reasoning, using few-shot prompting to ensure precise task separation."
- Algorithm 1, line 3: "Use LLM to decompose Q into subproblems P={p1,...,pn}".

Execution meaning:
- Decomposition is LLM-driven, not rule-only.
- The decomposition target is a set of smaller subproblems that can later be retrieved/resolved.
- The paper frames this as a runtime inference step, not offline preprocessing.

Invariant:
- Paper-faithful LogicRAG requires explicit subproblem generation before retrieval scheduling.

## 2) How dependencies are represented in the DAG

Paper text:
- "Each node in the DAG represents a subproblem, while edges encode the directional dependencies required for reasoning."
- "Let Q denote the input query, decomposed into a set of subproblems P={p1,...,pn}, where each pi corresponds to a node vi in a DAG G=(V,E)."
- "The directed edge set E ⊆ V × V encodes logical dependencies among them."
- "Edges are inferred by the LLM based on logical precedence among subproblems, and the resulting graph is verified via topological sorting to enforce acyclicity."

Execution meaning:
- Nodes are subproblems.
- Directed edges mean "this subproblem depends on that predecessor's answer/context first".
- Dependency modeling is semantic/logical precedence, not document citation links or corpus graph edges.
- Acyclicity is enforced after generation.

Invariant:
- In the paper, the DAG models reasoning dependencies between subproblems, not relationships among documents/chunks.

## 3) Whether insufficient retrieval triggers dynamic graph augmentation

Yes.

Paper text:
- "Dynamic Adaptation: If retrieval for a node yields insufficient context, the LLM dynamically augments the DAG by adding new subproblems and updating dependencies."
- Algorithm 1, lines 16-19:
  - "if LLM identifies a new unresolved subproblem p_{n+1} then"
  - "Add new node v_{n+1} to V and edge(s) to E if needed"
  - "Append p_{n+1} as a new rank after the current rank sequence"

Execution meaning:
- Dynamic expansion is part of the specified algorithm, not just discussion prose.
- The trigger is insufficient context / unresolved need discovered during execution.
- Expansion is additive: new node(s) plus new dependencies, appended after the current rank sequence.

Invariant:
- Paper-faithful LogicRAG is not a fixed one-shot plan; the DAG may grow during inference.

## 4) How topological ordering is used operationally

Paper text:
- "LogicRAG then linearizes the graph using topological sort, so that subproblems can be addressed in a logically consistent order."
- "Step 1: Topological Sort... yields an ordered sequence of subproblems <p(1), p(2), ..., p(n)>, in which every subproblem appears after all its dependencies."
- "This linearization respects the logical flow of reasoning and allows sequential resolution of subproblems in a single forward pass."
- Algorithm 1, line 6: "Topologically sort G to obtain ranks rank(pi) for all pi".

Execution meaning:
- Topological order is not just analysis; it becomes the scheduler.
- It converts dependency-constrained reasoning into a forward pass over ranks / sorted nodes.
- Retrieval for a node is conditioned on resolved parent outputs; the paper gives:
  - `C(i) = R(p(i) | previous parent results)`
  - `a(i) = f_RAG(p(i), C(i))`
- The purpose is to remove recursive/circular resolution and make the process tractable.

Invariant:
- Paper-faithful execution resolves only after dependencies are available; batching all subqueries flatly would violate the operational contract.

## 5) What graph pruning means in the paper

Paper text:
- Section title: "Graph pruning via unified subquery generation".
- "To reduce redundant retrievals and improve efficiency, we merge subproblems with the same topological rank into a single unified query."
- For same-rank set `S(i)`, the paper constructs `q_uni(i) = Merge(S(i))` and performs a single retrieval.
- "The process then advances to the next topological rank, skipping individual subproblem iterations."

Execution meaning:
- Graph pruning does not mean deleting low-score nodes globally.
- It specifically means consolidating semantically related / loosely coupled same-rank subproblems into one merged retrieval step.
- The payoff is fewer retrieval operations, shorter reasoning chains, and more consistent evidence across related subproblems.

Invariant:
- In this paper, graph pruning = same-rank unified query generation and shared retrieval, not arbitrary node dropping.

## 6) What context pruning / rolling memory means in the paper

Paper text:
- "Context pruning uses LLM-based summarization to maintain a rolling memory of relevant information."
- "We maintain a compressed memory state - a single text string that serves as a rolling summary of the most salient facts retrieved so far."
- After each subproblem is resolved, "its retrieved context and answer ai are distilled via LLM-based summarization" and merged into memory.
- Equation 3: `Mem(i) = Summarize(Mem(i-1) ∪ R(p(i)))`.
- Algorithm 1, lines 7 and 12 initialize and update rolling memory.

Execution meaning:
- Rolling memory is a summarized text state, not full-context accumulation.
- Each step summarizes newly retrieved chunks relative to the query and prior memory.
- The memory is the context carrier across steps, intended to suppress context bloat and irrelevant details.

Invariant:
- Paper-faithful context propagation is summary-based and lossy by design; passing all raw retrieved chunks forward is not what the paper describes.

## 7) How the paper avoids hesitation / repeated subquery generation

Paper text:
- The paper says iterative agentic RAG can suffer a "hesitation" issue where the model repeatedly generates similar subqueries.
- It compares:
  1. sampling with replacement: keep the current batch after generating the unified subquery,
  2. sampling without replacement: remove the current batch and force progression.
- The paper reports the first strategy is "prone to stalling at intermediate stages, producing near-duplicate subqueries until the maximum number of retrieval iterations is reached."
- It adopts sampling without replacement as default because it lowers token cost while keeping comparable answer quality.

Execution meaning:
- The anti-hesitation mechanism is scheduling/control-flow, not merely a prompt warning.
- Once a same-rank batch is processed, the system advances instead of letting the model keep revisiting it.

Invariant:
- Paper-faithful LogicRAG should enforce forward progress after a batch/rank is processed.

## 8) What the paper says about top-k vs token cost

Paper text from the "Efficiency-Effectiveness Trade-off of Top-k" section:
- Accuracy generally improves as k increases.
- Gains diminish beyond top-3 or top-5.
- On 2WikiMQA, gains from top-10 to top-20 are marginal.
- On MuSiQue, token cost exceeds 6000 tokens at top-20.
- Therefore, "moderate values (e.g., k=3 or k=5) offer the best balance between effectiveness and efficiency."
- The section explicitly links this to context pruning with rolling memory to avoid incremental context accumulation.

Execution meaning:
- The paper does not argue for maximizing k blindly.
- It treats top-k as a Pareto trade-off knob.
- Moderate retrieval breadth plus pruning is the intended efficient operating regime.

Invariant:
- Paper-faithful reproduction should report both answer quality and token cost, and should not assume larger k is always preferable.

## Additional implementation-relevant invariants

1. No pre-built graph:
   - LogicRAG's main claim is query-time adaptive reasoning structure without offline graph construction.

2. DAG is over subproblems, not corpus entities:
   - The graph organizes reasoning order, not a knowledge-base graph.

3. Execution is rank-wise:
   - Algorithm 1 processes each topological rank, merges same-rank nodes into a unified query, retrieves once, updates memory, then resolves each member subproblem.

4. Dynamic growth is appended after the current rank sequence:
   - The paper's pseudocode appends unresolved new subproblems after the current sequence rather than rebuilding the whole schedule from scratch.

5. Query-time efficiency comparisons exclude offline graph-build cost for graph baselines:
   - The paper separately argues those preprocessing costs are large and LogicRAG avoids them.

6. Retrieval in the paper is embedding-backed:
   - Algorithm 1 explicitly includes embedding model `θ`; any BM25-only adaptation is a local project adaptation, not the paper's native retrieval contract.

## Practical "do not overclaim" notes for AFAC2026

- If we keep only decomposition plus rank-wise execution, that is partial LogicRAG.
- If we skip same-rank unified-query graph pruning, we are no longer implementing the paper's pruning definition.
- If we pass full accumulated evidence instead of summarized rolling memory, we are not following the paper's context pruning contract.
- If we freeze the initial DAG and never allow augmentation when retrieval is insufficient, we are simplifying away an explicit part of the algorithm.
- If we replace the embedding retriever with BM25/RRF, we should call that an adaptation, not a paper-faithful reproduction.

## Extracted invariants list

- subproblems are LLM-generated at inference time
- dependencies are directed precedence edges between subproblems
- DAG must be acyclic and is checked by topological sort
- execution order is topological / rank-wise
- retrieval for downstream nodes is conditioned on resolved parent outputs
- insufficient context can add new nodes and dependencies dynamically
- graph pruning means same-rank merge into unified retrieval
- context pruning means rolling summarized memory, not raw context carry-forward
- anti-hesitation control is sampling without replacement / forced progression
- top-k has diminishing returns; k=3 or 5 is the paper's practical sweet spot
- paper retrieval stack is embedding-based, not lexical-only
