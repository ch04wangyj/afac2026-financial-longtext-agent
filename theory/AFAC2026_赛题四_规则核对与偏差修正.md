## 1. 官方规则要点

| 规则点 | 对工程的要求 |
|--------|--------------|
| 只允许 Qwen 系列模型 API 推理 | 所有在线推理、摘要、压缩、自检、重排序如果调用模型，都必须是 Qwen，并统计 Token |
| 禁止 embedding 模型 | 不接向量索引、不接 BGE/ColBERT/SPLADE/semantic chunking |
| 禁止 Qwen 以外开源/闭源模型 | 不接 LLMLingua 小模型、Cross-Encoder、VLM OCR 推理模型等作为默认链路 |
| 不修改基座模型参数 | 不微调、不 LoRA |
| 文档离线解析 Token 不计入 | PyMuPDF、pdfplumber、规则抽取、结构化字段可以离线做 |
| `answer.csv` 必须统计全部模型调用 Token | Qwen 子问题生成、压缩、自检、答案生成都要合并进 Token |
| A 榜有 `doc_ids`，B 榜无 `doc_ids` | A 榜可用于开发验证；B 榜必须先做文档级盲搜 |
| 多选题完全匹配 | 必须逐选项判断和排序去重，漏选/错选都失败 |

## 3. 发现的 bug

### Bug 1：置信度解析失败

现象：

- Qwen 输出：

```json
{"answer":"A","confidence":0.9,"reason":"..."}
```

但外层被 Markdown fence 包住：

```text
```json
{...}
```
```

- `parse_answer()` 能解析 fenced JSON，所以答案是 `A`。
- `Solver._extract_confidence()` 原来只接受裸 JSON，导致保存的 `confidence=0.0`。

修复：

- 在 `agent/reasoning/answer_parser.py` 中新增 `extract_json_object()`。
- `parse_answer()` 和 `Solver._extract_confidence()` 复用同一 JSON 提取逻辑。
- 新增单元测试覆盖 fenced JSON confidence。

验证：

```text
old_saved_confidence=0.0
fixed_parser_confidence=0.9
python -m unittest discover -s tests -> 17 tests OK
python -m compileall agent scripts tests -> OK
```

说明：旧的 `outputs/live_smoke/answer_results.jsonl` 是修复前保存的产物，因此里面仍是 `confidence=0.0`；修复后同类输出会解析为 0.9。`answer.csv` 不包含 confidence，所以该 bug 不影响本次提交格式，但会影响后续低置信路由和自检。

### Bug 2：多文档题证据压缩只保留了第一份文档

现象：

- `fc_a_001` 的 `doc_ids=['text01','text02']`，题目要求比较两份债券募集说明书。
- 真实运行中压缩后的 8 条 evidence 全部来自 `text01`。
- Qwen 原始回复明确说“未提供第二份文档的相关信息”，因此只能输出 `A`，无法判断 B/D。

原因：

- `RuleEvidenceCompressor` 只按全局分数排序取 Top-K。
- 第一份文档因与题干/选项 A/C 高度匹配，占据全部 evidence 名额。
- 多文档题没有“每个候选文档至少保留一条证据”的配额。

修复：

- `RuleEvidenceCompressor.compress()` 增加多文档覆盖策略：当 `question.doc_ids` 多于 1 个时，先按 `doc_ids` 顺序为每个文档保留最高分证据，再填充剩余 Top-K。
- 新增 `tests/test_rule_filter.py` 覆盖该场景。

验证：

```text
dry-run fc_a_001 evidence_doc_ids = ['text01', 'text02', 'text01', ...]
python -m unittest discover -s tests -> 18 tests OK
python -m compileall agent scripts tests -> OK
```

## 4. 当前偏离点

| 偏离点 | 风险 | 修正方向 |
|--------|------|----------|
| 文档中过多讨论 GraphRAG/LightRAG/HippoRAG/RAPTOR 等通用框架 | 容易误导为要接 embedding、聚类、小模型或复杂图系统 | 全部降级为“不可直接采用/只借鉴思想”，默认只保留无 embedding 的 BM25/RRF/规则结构化查询 |
| 真实调用默认 `enable_thinking=true`，1 题 completion_tokens 达 846 | Token 成本偏高，不符合“尽可能少 Token”目标 | V1 默认应支持非 thinking 快速答题；低置信/难题再启用 thinking 或 qwen3.7-max |
| 当前 A 榜链路默认按 `doc_ids` 限定检索 | A 榜可用，B 榜不可用 | 增加文档级盲搜阶段，B 榜先选候选文档再 chunk 检索 |
| 真实运行只跑 1 题 | 不能代表准确率 | 需要跑 5 领域各 1 题 smoke，再做人工核查或提交 A 榜获取真实准确率 |
| 当前推理是整体判断 | 多选题容易漏选/误选 | V2 必须逐选项检索和判断 |

## 5. 收敛后的工程路线

### 默认合规主线

1. 离线解析：PyMuPDF/pdfplumber/规则 OCR 预留，不使用推理模型。
2. 索引：BM25 + char n-gram + jieba + 字段索引，不使用 embedding。
3. 检索：`doc_first_bm25f_expansion` 主路；`question_options` 降级为 legacy sparse baseline，`rule_multi_rrf` 作为补召回/实验对照。
4. 压缩：规则证据压缩，保留条款号、页码、数字、日期、表格行。
5. 推理：Qwen-only，所有调用统计 Token。
6. 答案：严格字母解析，多选排序去重。

### 低置信升级

只有当检索覆盖不足、答案非法、证据冲突、多选题或财报计算题时，才升级：

- Qwen 子问题生成；
- thinking 开启；
- qwen3.7-max 复核；
- CoVe/IRCoT-lite 自检。

### 需要暂缓的路线

- 原版 GraphRAG、LightRAG、HippoRAG、RAPTOR、Self-RAG、OpenSPG/KAG 不进入默认实现。
- 只允许保留不依赖 embedding/非 Qwen 模型的 lite 思想：实体共现、规则子查询、RRF 融合、Python KVP。
