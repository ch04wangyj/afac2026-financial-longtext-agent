| 路线 | 处理 | 原因 |
|------|------|------|
| Embedding / SPLADE / ColBERT / BGE reranker | ❌ 删除 | 违反或高度接近违反“禁止 embedding / 仅限 Qwen 系列模型”约束 |

## 2. BM25 检索理论

### 2.1 概率检索框架下的 BM25

BM25 由 Robertson、Spärck Jones 等人基于**概率检索框架 (Probabilistic Relevance Framework)** 从二值独立模型 (BIM) 发展而来。

$$\text{BM25}(D,Q) = \sum_{t \in Q} IDF(t) \times \frac{f_{t,D} \cdot (k_1 + 1)}{f_{t,D} + k_1 \cdot \left(1 - b + b \cdot \frac{|D|}{avgDL}\right)}$$

**三个核心信号**：

| 信号 | 公式组件 | 对金融文本的意义 |
|------|----------|-----------------|
| **IDF**：稀有词权重高 | `log((N-n+0.5)/(n+0.5)+1)` | "身故保险金"在全部文档中出现很少→IDF极高→精确命中 |
| **TF 饱和 (k₁)**：词频非线性 | 分子 `f×(k₁+1)`，分母 `f + k₁×(…)` | "营业收入"在财报中出现50次→饱和处理，避免过度加权 |
| **文档长度归一化 (b)** | `(1-b) + b×|D|/avgDL` | 表格段落极短，法规条文中等，研报很长→分别归一化 |

### 2.2 为什么 BM25 完美匹配本赛题约束

1. **完全无 Embedding**：BM25 只需词法和统计信息（TF、DF、DL），零模型依赖
2. **对精确术语天生敏感**：金融/法律文本中的精确术语（产品名、法条号、指标名）在词法层面唯一，BM25 直接捕获
3. **离线构建，在线零 Token**：索引在预处理阶段构建，每次检索 0 Token
4. **可解释**：可以追溯每个文档的分数来源——哪个词贡献了多少

### 2.3 领域自适应参数

```python
BM25_PARAMS = {
    # k1: 词频饱和 | b: 长度归一化
    "insurance":           {"k1": 1.5, "b": 0.75},  # ~20页条款，适中
    "regulatory":          {"k1": 1.2, "b": 0.50},  # 法条精炼，低归一化
    "financial_contracts": {"k1": 1.5, "b": 0.75},  # 300+页，均衡
    "financial_reports":   {"k1": 1.8, "b": 0.30},  # 表格多，极低归一化
    "research":            {"k1": 1.2, "b": 0.75},  # 长叙述，标准
}
```

### 2.4 伪相关反馈 (Pseudo Relevance Feedback)

```
Round 1: BM25 检索 Top-10
    → 从 Top-10 中提取高 IDF 词作为扩展词
Round 2: 用 原始查询 + 扩展词 重新检索
    → 显著提升召回（SIGIR 2024 实验证实超越纯 BM25）
```

**论文支撑**：
- ⭐ *"Old IR Methods Meet RAG"* — Huly et al., SIGIR 2024 — BM25 **超越**稠密检索；词邻近度 MRF 超越 BM25；PRF 超越 MRF
- ⭐ *IterKey* — Hayashi et al., 2025 (arXiv:2505.08450) — LLM 迭代生成关键词增强 BM25，达 +5~20% 准确率

## 3. 提示词压缩理论

### 3.1 "Lost in the Middle" 与位置工程

LLM 对长上下文的注意力呈 **U 形分布**：开头和结尾信息被关注，中间信息被忽视。因此：
- 最重要的证据放在 prompt **开头或结尾**
- 压缩的目标不只是省 Token，更是**信息密度最大化**
- 一篇 3000 token 的段落，LLM 实际有效利用的可能只有 1000 token

**论文支撑**：
- *"Lost in the Middle"* — Liu et al., 2024
- *Position Engineering* — Jiang et al., 2024

### 3.2 抽取式 vs 摘要式压缩

| 维度 | 抽取式 (Extractive) | 摘要式 (Abstractive) |
|------|---------------------|---------------------|
| 原理 | 按重要性分数保留原文句子 | LLM 重新生成压缩文本 |
| 数值/条款保留 | ✅ 完全保留（因为是原文） | ❌ 可能丢失精确数值 |
| Token 成本 | 0（规则/词法）或高风险小模型 | 500-2000（Qwen 调用） |
| 本赛题适用性 | ⭐⭐⭐⭐⭐ **首选** | ⭐⭐⭐ 辅助 |

**Jha et al. (2024) 标准化评测结论**：抽取式压缩全面优于其他方法，可实现 10× 压缩而精度损失最小。

### 3.3 小模型压缩路线的规则风险

| 版本 | 核心机制 | 工程收益 | 当前处理 |
|------|----------|----------|----------|
| **LLMLingua / LongLLMLingua** | 小 LM perplexity 或问题感知过滤 | 压缩快，能降 Token | ⚠️ 不作为默认方案，避免非 Qwen 模型争议 |
| **LLMLingua-2** | BERT 二分类，数据蒸馏自 GPT-4 | 压缩更快 | ⚠️ 仅做本地实验，不进入默认提交链路 |
| **DSPC / CPC / FrugalPrompt** | 新型句子选择或对比学习编码器 | 思路可借鉴 | ⚠️ 实现为规则句子选择，不加载额外模型 |

> 默认策略：采用**规则抽取式压缩**，用条款号、标题层级、表格行、数值/日期/比例、题干关键词和选项关键词打分，保留原文证据。这样零 Token、可解释、可复现，也规避非 Qwen 模型风险。

## 4. Agent 记忆管理理论

### 4.1 记忆六操作 ("Rethinking Memory in AI", 2025)

| 操作 | 本赛题对应 | 实现 |
|------|-----------|------|
| **Consolidation** | 多段落证据 → 统一判断 | Phase 4 逐选项推理 |
| **Indexing** | 条款编号 → 内容 hash map | 离线预计算字段索引 |
| **Retrieval** | BM25 + 字段精确查找 | Phase 2 |
| **Compression** | 规则抽取式压缩 + KVP/表格 JSON | Phase 3 |
| **Forgetting** | 已判断选项的证据释放 | Phase 5 |
| **Updating** | 更强证据替换弱证据 | 迭代检索时 |

### 4.2 上下文毒化与上下文分心

- **Context Poisoning**：上下文中的错误被反复引用，恶化所有后续推理
- **Context Distraction**：过多无关信息导致模型忽略关键信号

**对本赛题的启示**：不要让 Qwen "阅读全文"。必须精准投喂与当前选项相关的 500-800 token 证据片段。

### A.3 多索引构建

```python
class DocumentIndex:
    """离线构建，全程不计Token"""
    def __init__(self, domain):
        self.bm25 = None             # BM25 词法检索（主索引）
        self.clause_index = {}       # "第47条" → 完整条文内容
        self.entity_index = {}       # "平安智盈金生" → 所在chunks
        self.numeric_index = []      # [(value, unit, chunk_id), ...] 数值范围
        self.chunks = []             # 所有文本块
        self.fact_cards = {}         # 每个文档的结构化摘要

    def build(self, parsed_docs):
        # 1. 分块
        self.chunks = chunk_all(parsed_docs)

        # 2. BM25（中文用字符n-gram或jieba+自定义词典）
        self.bm25 = BM25Index(self.chunks, **BM25_PARAMS[self.domain])

        # 3. 条款/法条精确索引
        for chunk in self.chunks:
            for clause_id in extract_clause_ids(chunk.text):
                self.clause_index[clause_id] = chunk

        # 4. 实体精确索引 (Aho-Corasick)
        for chunk in self.chunks:
            for entity in extract_entities(chunk.text):
                self.entity_index.setdefault(entity, []).append(chunk)

        # 5. 数值范围索引
        for chunk in self.chunks:
            for num in extract_numbers(chunk.text):
                self.numeric_index.append(num)

        # 6. DocProfile / KVP (规则生成的结构化画像)
        for doc_id, doc_text in parsed_docs.items():
            self.doc_profiles[doc_id] = self._build_doc_profile(doc_text)

    def _build_doc_profile(self, doc_text):
        """规则抽取：不调用LLM，保留标题、实体、条款号、关键数值和日期"""
        return {
            "company": "平安智盈金生",
            "key_clauses": {"6.1": "犹豫期20日全额退保", "3.4": "投资组合账户每年结算"},
            "key_numbers": {"退保费用率_第5年": "1%", "退保费用率_第7年": "0%"},
        }
```

## 模块 D. 推理（核心，Token 最大开销）

### D.1 逐选项独立判断（多选题的关键策略）

```python
def answer_multi_choice(question, options, evidence):
    """多选题：把 multi 变成 4 个独立的 tf"""
    results = {}
    for option_letter in ["A", "B", "C", "D"]:
        # 每个选项独立判断，互不干扰，避免上下文混淆
        prompt = f"""仅判断以下选项是否成立。只输出"成立"或"不成立"，并给出原因和证据引用。

证据：{compress_for_option(evidence, options[option_letter])}

题干：{question}
选项{option_letter}：{options[option_letter]}

请判断："""
        result = qwen_call(prompt, max_tokens=300)
        results[option_letter] = parse_verdict(result)

    # 汇总 + 去重排序
    correct = sorted([o for o, v in results.items() if v])
    return "".join(correct)  # e.g., "AC"
```

**为什么这样做？**
- 多选题的 4 个选项之间可能相互干扰——LLM 看到 A 的判断后可能影响对 B 的判断
- 独立判断每道选项 → 消除上下文污染
- 各选项 Token 独立计算（4×300=1200 tokens），质量优先

### D.2 Calculator Agent（数值计算）

```python
def calculator_reasoning(question, evidence):
    """CoT 推理 + Python 计算执行"""
    # Step 1: Qwen CoT 提取公式和数值
    cot = qwen_call(f"""
    请逐步分析，最后给出计算公式和代入的数值。
    题目：{question}
    证据：{evidence}
    输出格式：
    步骤1: ...
    步骤2: ...
    公式: 身故保险金 = max(已交保费, 现金价值, 保单账户价值)
    代入: max(1000000, 800000, 900000)
    """, max_tokens=500)

    # Step 2: Python 执行精确计算（不耗推理 Token）
    result = safe_execute(cot["公式"], cot["代入"])

    # Step 3: 结果验证
    return result
```

**为什么 Calculator Agent 很重要？** 保险和财报的计算题涉及百万级金额的精确比较，LLM 心算极易出错。Tan et al. (2025) 实验表明：CoT+Calculator 消除 88% 算术错误。

```python
def format_answer(llm_output, answer_format):
    # 提取所有 A-D 字母
    letters = re.findall(r'[A-D]', llm_output.upper())

    if answer_format == "mcq":
        return letters[0] if letters else ""
    elif answer_format == "multi":
        return "".join(sorted(set(letters)))
    elif answer_format == "tf":
        return letters[0] if letters else ""

def validate_answer(answer, answer_format):
    if answer_format == "mcq":
        return len(answer) == 1 and answer in "ABCD"
    elif answer_format == "multi":
        return all(c in "ABCD" for c in answer) and answer == "".join(sorted(answer))
    elif answer_format == "tf":
        return answer in "AB"
```
