# 简历案例：金融长文本 RAG Agent

## 一句话项目描述

在 Qwen-only、禁止 embedding、500 万 Token 预算的比赛约束下，设计金融长文本 RAG Agent，覆盖保险、监管、合同、财报和研报五个领域；通过结构化 PDF 解析、原子谓词检索、PageIndex-lite 导航和逐选项证据闭环，将官网得分从 `58.1679` 提升到 `83.33`。

## 可直接用于简历的版本

**金融长文本 RAG Agent｜核心开发**

- 设计无 embedding 的 BM25F + PageIndex-lite 两阶段检索，联合标题、章节、条款号、数值、日期、表头与单位字段，处理 5 类金融长文档和 100 道多文档选择题。
- 基于 PyMuPDF 坐标与矢量线实现确定性 PDF 版面解析，生成 32,663 个版面块和 14,302 个结构化表格行，降低跨列、跨年和单位错配。
- 将粗粒度页面重构为 38,544 个原子谓词块，分离“候选支持查询”和“谓词真实值查询”，并用支持/反证证据覆盖替代模型自报置信度。
- 建立产品/公司实体绑定、逐选项 `true/false/uncertain` 裁决和保守发布机制；54 个候选变化只接受 14 个直接原文闭环，官网验证净增 12 题。
- 正确率由 `62%` 提升到 `85%`，综合分由 `58.1679` 提升到 `83.33`；在线 Token 从 103 万降到 32.6 万，减少约 `68.3%`。
- 在 V6/V7 中实现选项级证据契约、财务事实账本、来源约束的跨题事实审计和题干范围门禁；V6 官网净增 3 题，V7 全量负向实验则保留为消融证据。

## STAR 叙事

### Situation

金融文档长、表格密集、同类产品条款相似。赛题禁止 embedding 和非 Qwen 模型，同时将答案准确率与在线 Token 共同计分。初版系统能召回相关页面，但经常发生表格列错配、错误选项牵引检索、跨产品条款串用和多选漏选。

### Task

在不修改基座模型参数、不使用向量模型的条件下，提高 100 题准确率，并确保所有模型调用可统计、答案可追溯、流水线可复现。

### Action

1. 先把财报表格转换为行级事实并建立答案门禁。
2. 用文档级穷举验证召回上限，发现正确率提高但 Token 和综合分恶化。
3. 将页面切分为原子谓词块，用真实值查询和反证选择提高证据密度。
4. 恢复 PDF 表头、单位、跨页续表和视觉行结构。
5. 引入结构导航和文档实体绑定，逐选项判真并用直接证据白名单控制发布。
6. 将“检索相关性”升级为“证据充分性”：检查必需文档、谓词、数值端点、冲突和财务口径，并为近重复题建立只审计不自动改答案的事实图。

### Result

- 官网得分：`58.1679 → 83.33`。
- 正确率：`62% → 85%`。
- 在线 Token：`1,030,141 → 326,076`。
- V4 到 V5：Token 仅增加约 1%，净增 12 道正确题。
- 工程质量：提交格式、Token 守恒、题目覆盖、答案合法性和题面版本均有自动门禁。

## 为什么不选其他技术栈

### 为什么不是 embedding RAG

赛题明确禁止 embedding。系统因此使用 BM25F、字符/词混合 tokenizer、精确数值扫描和结构字段索引。这个约束也强化了可解释性：每个召回分数都能追溯到具体词法或结构字段。

### 为什么不是 GraphRAG

A 榜已经提供候选 `doc_ids`，主要问题是文档内部条款、表格事实和跨文档比较，而不是开放域实体发现。全局图构建会增加抽取误差和维护成本，对当前局部证据定位收益有限。项目只吸收实体绑定和局部结构关系，不构建全局知识图。

### 为什么不是全题 LogicRAG / PoT

真实实验中，全题 PoT、thinking、自验证和 Judge 将 Token 从约 31 万提高到约 64 万，并造成 52 题答案漂移。问题根因是证据身份和覆盖，不是推理步骤不够长。因此改为逐选项证据闭环和确定性答案组装。

### 为什么不是全量 OCR / VLM

题目主要 PDF 有可用文字层。坐标和矢量线解析零模型 Token、结果稳定、便于测试；旧 Qwen-VL 候选覆盖后官网分数下降。VLM 只适合未来对扫描页按缺字率触发，不应作为全量默认解析器。

### 为什么保留 BM25

BM25 本身不是最终方案，但它适合条款号、专名、金额、百分比和日期。项目通过 BM25F 字段化、原子 chunk、谓词查询、结构导航和证据精排补足其语义局限，而不是把全部问题归因于稀疏检索。

## 面试可展开的工程问题

### 如何证明提升来自方法而不是随机模型波动

- 所有版本保存完整 `answer_results.jsonl`、证据和逐题 Token。
- 发布候选与官方基线做答案 diff。
- 未经直接原文复核的答案变化默认回退。
- 官网分数用于最终验证；本地 3 题开发集只做回归门禁，不宣称代表总体准确率。

### 如何处理多选题

每个选项独立检索和判真，输出 `true/false/uncertain`。最终答案由程序按字母顺序组装，避免自由文本答案与逐项分析冲突；任何 `uncertain` 不会自动变成正确选项。

### 如何处理表格

先恢复视觉行和列结构，再把表名、单位、年度层级表头复制到每个数据行。检索时表格行作为独立原子证据，并对指标、年份、值和单位做完整性检查。

### 如何控制回归

结构导航只增加候选，不删除全局 BM25F 命中；新版候选答案默认回退旧版；人工复核配置记录答案、原因和直接原文依据。该设计允许独立消融每个模块。

### V6/V7 体现了什么工程判断

V6 全量候选有 41 题变化，最终只发布 6 个直接原文闭环，官网验证净增 3 题。V7 初版把双层适用性判断用于全题，造成 42 题漂移；复盘后只让 6 道显式集合题触发，6/6 稳定复现官方基线。这体现了离线实验、失败回滚、发布门禁和线上反馈之间的边界管理。

## 90 分目标

在 `326,076` Token 下，92 道正确题预计得分约 `90.2001`。下一阶段需要相对 V6 净增 7 题，计划按以下门禁推进：

1. 扩充带页码、题面 SHA-1、指标和单位的直接证据开发集。
2. 为保险免赔额、赔付顺序、合同罚息和财务比率建立确定性计算器。
3. 建立表格 `metric/year/value/unit` schema 对齐和跨页表头一致性测试。
4. 对“全部、均、仅、不得、除外”等全称和否定断言执行覆盖检查。
5. 每次只提交有直接证据闭环的答案变化，并报告净变化与 Token 变化。

90 分是工程目标，不应在未知官方标签下承诺必然达到；可承诺的是使用上述门禁持续减少不可解释的答案漂移，并以官网结果验证每轮增益。
